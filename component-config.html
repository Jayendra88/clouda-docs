Architecture · Component configuration

Component configuration, per component and per tenant

Break the single ~11.8k-line sections.json into small *.section.json files that live next to each component, add a tenant slug field, and combine them at build into exactly the shape the composer already consumes — so nothing downstream breaks and each tenant sees only the components it should.

Proposed — awaiting approval. Nothing implemented yet.

Today: one giant sections.json

A single configuration file — around 11,800 lines — lists all of the roughly 64 components as a flat array of { name, schema } entries. The composer palette in sectionRenderer.tsx reads that file flat and renders every entry to every user.

Hard to maintain

Every component change edits one enormous shared file — merge conflicts, no ownership, easy to break unrelated entries.

No categorisation

A flat array: no categories, tags, or status. The palette can only show one long undifferentiated list.

Identical for every tenant

There is no tenant field, so every tenant is offered the exact same set of components — no way to enable or restrict.

glue-components/cms/schema.json (today, abridged) — imported by the composer via @clouda-inc/glue-components/dist/cms/schema.jsonjson
// a bare, top-level ARRAY — the composer does (schemas as any[]).find(s => s.name === …)
[
  { "name": "HeaderMain", "schema": { /* … */ } },
  { "name": "HeroBanner", "schema": { /* … */ } }
  /* … ~64 entries, ~11.8k lines, no category / tenant … */
]

Per-component config files

Replace the monolith with a small config co-located with each component: a ComponentName.section.json that sits beside its .tsx. Ownership moves to the component, diffs stay local, and each file can carry richer metadata: category, tags, status, and a tenants/visibility field.

packages/components/src/header-main/HeaderMain.section.jsonjson
{
  "name": "HeaderMain",
  "category": "navigation",
  "tags": ["header", "nav", "sticky"],
  "status": "stable",
  "visibility": "global",          // "global" | ["acme","brandb"] | "acme"
  "schema": {
    "logoUrl": { "type": "string", "label": "Logo URL" },
    "sticky":  { "type": "boolean", "default": true }
  }
}
Naming is the contract. The file basename (HeaderMain) must equal the name field and match its folder — validation enforces this so the build and the composer never disagree about what a component is called.

Combine at build

A build step globs every *.section.json, validates them, and combines them into a single sections.json that keeps the exact shape the composer already consumes — so sectionRenderer.tsx needs no change to keep working. Alongside it, the step emits a richer manifest.json carrying the metadata (category, tags, status, visibility) the new palette uses. Wire the script into the existing Rollup build so it runs on every package build.

Many co-located *.section.json files → one aggregate step → sections.json (unchanged shape) + manifest.json (metadata).
packages/components/build/aggregate-sections.jsjs
import { globSync } from "glob";
import { readFileSync, writeFileSync } from "node:fs";

const files = globSync("src/**/*.section.json");
const seen = new Map();
const sections = [];   // composer shape: { name, schema }
const manifest = [];   // rich metadata

for (const file of files) {
  const cfg = JSON.parse(readFileSync(file, "utf8"));

  // validate name uniqueness
  if (seen.has(cfg.name)) {
    throw new Error(`Duplicate component name "${cfg.name}" in ${file}`);
  }
  seen.set(cfg.name, file);

  sections.push({ name: cfg.name, schema: cfg.schema });
  manifest.push({
    name: cfg.name,
    category: cfg.category ?? "uncategorised",
    tags: cfg.tags ?? [],
    status: cfg.status ?? "stable",
    visibility: cfg.visibility ?? "global",   // safe default
  });
}

// bare array — byte/shape-compatible with what sectionRenderer.tsx's (schemas as any[]).find(...) expects
writeFileSync("dist/sections.json", JSON.stringify(sections, null, 2));
// the new manifest is our own artifact — an object wrapper is fine (read as manifest.manifest below)
writeFileSync("dist/manifest.json", JSON.stringify({ manifest }, null, 2));

Tenant slug field: enable/restrict per tenant

Each *.section.json (or, later, a component_registry row) carries a visibility field with one of three shapes:

Global

"global" — offered to every tenant. This is also the default when the field is missing.

List of slugs

["acme","brandb"] — offered only to the listed tenant slugs.

Single tenant

"acme" — offered to exactly one tenant slug.

Missing means global. A component with no visibility is treated as "global", so existing components keep appearing for everyone — the safe default preserves today's behaviour exactly.
Not a code fork. A "tenant-specific component" is a normal shared component whose visibility is limited to one or more slugs. There is only one codebase; visibility just filters which slugs may pick it in the palette.

The composer filters the palette for the active tenant using this rule:

glue-composer/src/lib/getVisibleComponents.tsts
import manifest from "@clouda-inc/glue-components/manifest.json";

type Visibility = "global" | "string" | "string"[];

export function getVisibleComponents(tenantSlug: "string") {
  return manifest.manifest.filter((c) => {
    const v: Visibility = c.visibility ?? "global";   // missing => global
    if (v === "global") return true;
    if (Array.isArray(v)) return v.includes(tenantSlug);
    return v === tenantSlug;                          // single slug
  });
}

Validation

A lint / CI step guards the whole pipeline so a bad config fails the build rather than the composer at runtime:

  • Unique names — no two *.section.json files may declare the same name.
  • Folder ↔ name agreement — the file basename and its parent folder must match the name.
  • Allowed categories — category must be one of a known enum (e.g. navigation, hero, content, media, form, footer).
  • Byte-compatible output — the generated sections.json must stay structurally compatible with what sectionRenderer.tsx expects, so nothing downstream breaks.
Fail fast. Running validation in CI means a duplicate name, a mismatched folder, or an unknown category is caught in the pull request — never after deploy.

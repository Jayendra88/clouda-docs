Clouda Glue — Multi-Tenant Platform
Comprehensive Build Plan (v3)

The document is organized in three parts: Part A covers cross-cutting concerns (the seams where multi-repo systems typically fail), Part B covers each repo in dependency order, and Part C covers the concrete build/rollout sequence — the order the pieces get developed and how they connect as work progresses.

1. Executive summary

The platform is a multi-tenant e-commerce and content system split into a set of published packages plus one shared editor deployment. The shared packages carry no tenant-specific code. Each tenant gets its own repo, generated from a template, that is their deployed storefront — the same relationship a FastStore store has with @faststore/core. One additional application, the composer, is deployed once and serves every tenant as a visual CMS by loading each tenant's components dynamically at runtime via Module Federation.

The two hardest problems in a system like this are not architectural, they are integration seams:

  • Keeping the storefront's compiled code and the composer's dynamically-loaded view of that same tenant consistent, deploy after deploy.
  • Keeping contracts (component prop schemas, TypeScript types, database shapes) synchronized as they evolve.

Part A of this document addresses those seams explicitly. Parts B and C then specify each repo and the order to build them.

2. Goals & non-goals

Goals

  • Tenants edit only their own repo — components, services, theme — and never the shared packages or other tenants' code.
  • npm run dev in a tenant repo starts that tenant's actual storefront locally with full hot reload.
  • Tenants can use prebuilt components unmodified, restyle via tokens, or wrap them with extra markup/logic.
  • Non-developers can edit page content and theme in the composer without triggering a redeploy of any tenant's app.
  • The composer is one deployment serving every tenant, resolving each tenant's components and schemas dynamically.
  • Shared packages are versioned strictly by semver, tested once, and consumed identically by every tenant.

Non-goals

  • No shared multi-tenant storefront deployment. Each tenant's storefront is its own deployment.
  • No direct database access outside the glue-data interface, for any consumer.
  • No cross-tenant component sharing outside the shared packages — a tenant cannot import another tenant's repo.

4. Deploy atomicity: single source of truth per tenant version

Problem this addresses. The storefront is built from src/components/index.ts. The composer sees the tenant via src/federation/expose-components.ts. If these two artifacts are built at different commits, the composer offers components the storefront can't render, or renders props against a schema the storefront doesn't understand.

Design

  1. src/federation/expose-components.ts is a one-line re-export of src/components/index.ts. It is enforced by an ESLint rule in the tenant repo template — any deviation fails CI.
  2. The tenant CI pipeline is a single atomic job with three outputs from the same commit SHA: (a) the deployed storefront, (b) the remoteEntry.js bundle at cdn.clouda.io/tenants/<slug>/<sha>/, (c) a manifest describing what was built.
  3. tenant_registry records deployedCommit (SHA), remoteEntryUrl (points to <sha>/), remoteEntryIntegrity (SRI hash), and version (human-readable). Only publish-service writes this row — never CI directly.
  4. The storefront's app/layout.tsx logs a warning at boot if process.env.BUILD_COMMIT !== (await getTenantFromRegistry(slug)).deployedCommit. This catches drift within seconds of a broken deploy.
  5. Rollback is a first-class operation on publish-service: point tenant_registry at a prior <sha>/ that still exists on the CDN. Composer switches to the older components; the tenant separately redeploys the storefront at that SHA. The two steps do not have to happen atomically — the storefront and composer converge back to consistency once both are pointed at the same SHA.

5. Schema evolution & typed contracts

Problem this addresses. Each component ships a schema.json. Composer validates page props against it, generates the editor UI from it, and (implicitly) tenant devs write component code that reads matching props. If a tenant renames a prop, existing page documents in the DB still carry the old name — the storefront breaks on the next request.

Design — additive-only, with generated types

  1. Additive-only rule. Fields may be added or deprecated, never renamed or removed. A CI check diffs each PR's schemas against the last built version and fails on removal/rename.
  2. Deprecation flow. A field is marked "x-deprecated": true with an optional "x-replacedBy". Composer hides deprecated fields in the editor but still lets the storefront read them. A migration script (in glue-services) can be run out-of-band by the platform team to rewrite existing page docs — this is explicit, never automatic.
  3. Types are generated, not hand-written. A build step in the tenant repo runs json-schema-to-typescript against every schema.json and writes types to src/components/generated/props.d.ts. Tenant component code imports these types, so the component's TypeScript definition and the composer's validation contract cannot drift.
  4. Publish-time double-check. When composer saves a page, publishPage() validates props against schema (as before). Additionally, the storefront's render function tolerates unknown extra props (drops them) but logs a warning — this prevents a stale composer view from taking down the site during the window between a component removal and its rollout.
// package.json script the tenant template runs before every build/dev
"scripts": {
  "generate:types": "node scripts/schemas-to-types.mjs",
  "prebuild": "npm run generate:types",
  "predev":    "npm run generate:types"
}

6. Cache semantics & failure modes for tenant_registry

Problem this addresses. The plan says "cached, short TTL" without specifying where the cache lives, how invalidation works, or what happens when the registry is unreachable. Those decisions have real product consequences (how long after a marketer clicks Save do their edits appear).

Design

  1. Cache layers. Per-instance in-memory cache (LRU, 30-second TTL) in front of the tenant_registry DB. Not Redis — the registry is small enough that per-instance memory is fine and one fewer dependency to run.
  2. Eager invalidation. When composer writes themeOverrides for a tenant, it POSTs to the tenant's storefront at /_internal/invalidate with the tenant slug and an HMAC signature verifying it came from composer. The storefront invalidates its cache immediately and re-fetches on next request. Signature keys live in shared secrets between composer and the tenant's env.
  3. Product SLO. "Composer changes visible within 5 seconds under normal operation; within 30 seconds if webhook delivery fails." This is documented and testable.
  4. Stale-if-error. If a request comes in and the cache is empty and tenant_registry is unreachable, the storefront serves stale data from an on-disk fallback snapshot written each time the cache successfully refreshes. Better a slightly-stale page than a 500. The composer, in contrast, fails closed — an editor should not save against a stale registry view.
// packages/glue-data/src/tenant-registry-cache.ts
import LRU from 'lru-cache';
const cache = new LRU<string, TenantRow>({ max: 200, ttl: 30_000 });

export async function getTenantFromRegistry(slug: string): Promise<TenantRow> {
  const cached = cache.get(slug);
  if (cached) return cached;
  try {
    const row = await fetchFromDb(slug);
    cache.set(slug, row);
    await writeFallbackSnapshot(slug, row); // storefront-only
    return row;
  } catch (err) {
    const fallback = await readFallbackSnapshot(slug);
    if (fallback) { logStaleServed(slug, err); return fallback; }
    throw err;
  }
}

export function invalidateTenant(slug: string) { cache.delete(slug); }

// tenant-repo/app/_internal/invalidate/route.ts
import { invalidateTenant } from '@clouda-inc/glue-data';
import { verifyComposerHmac } from '@clouda-inc/glue-services';

export async function POST(req: Request) {
  if (!verifyComposerHmac(req)) return new Response('forbidden', { status: 403 });
  const { slug } = await req.json();
  invalidateTenant(slug);
  return Response.json({ ok: true });
}

7. The shared React constraint

Problem this addresses. Composer loads multiple tenants' remotes into one host at runtime. Module Federation's singleton behavior only works if every remote agrees on React's major version and shared config. This is a genuine limit on tenant independence and must be stated openly.

Design

  1. glue-composer's package.json declares the supported React major (currently 18). This is the platform's compatibility promise for the current cycle.
  2. Tenant repo template pins react and react-dom to the exact minor the platform team blesses (e.g. 18.3.1), not a caret range.
  3. The tenant CI includes a shared lint step from @clouda-inc/tenant-ci-config that fails the build if the pinned React version drifts from the platform's supported version.
  4. A React major upgrade is a coordinated platform event: composer + all tenant templates bump together. The bump-all mechanism from §24 exists partly to make this operable.
This is a real constraint on tenant autonomy. Document it in the tenant developer onboarding docs, not just this build plan.

8. The publish API — first-class contract

Problem this addresses. Multiple sections mention "the internal publish API." It is the single most security-sensitive integration point (tenant CI touches shared infrastructure) and was previously undefined. This section is its complete spec.

Endpoint

POST https://publish.clouda.io/v1/tenants/<slug>/versions

Auth

  1. Each tenant repo, at creation, is issued a signing keypair. The private key lives in that repo's CI secrets store, scoped to the repo.
  2. Requests carry a short-lived JWT signed with the private key: subject is the tenant slug, issued-at within the last 5 minutes, audience is publish.clouda.io.
  3. publish-service verifies the JWT against the public key registered in tenant_registry at repo-creation time. A mismatch or a signature over a different slug is rejected.

Request

{
  "commit": "9a7f2c1b...",
  "version": "v42",
  "remoteEntryUrl": "https://cdn.clouda.io/tenants/acme/9a7f2c1b/remoteEntry.js",
  "remoteEntryIntegrity": "sha384-...",   // SRI hash, computed by CI
  "glueCoreVersion": "2.3.1",
  "storefrontUrl": "https://acme.com",     // where the storefront was deployed
  "idempotencyKey": "9a7f2c1b-abc123"       // safe to retry
}

Behavior

  1. Idempotency: same key returns 200 with the prior result; no double-write.
  2. Verification: publish-service HEAD-checks remoteEntryUrl exists on the CDN and re-computes the SRI hash to confirm it matches. If either fails, respond 422 — the tenant's storefront deploy is not rolled back, but composer keeps pointing at the previous version.
  3. Registry update: single atomic write of deployedCommit, remoteEntryUrl, remoteEntryIntegrity, version, glueCoreVersion.
  4. Response: 200 with { ok: true, previous: {...} } so the tenant CI can log the previous state for its own rollback command.

Rollback

POST /v1/tenants/<slug>/rollback with { "toCommit": "..." }. Requires the same JWT auth. Rejects if the target commit's remoteEntry.js no longer exists on the CDN (retention keeps the last 10 versions).

Partial-failure story

If CDN upload succeeds but the publish-service call fails, the tenant CI retries with the same idempotency key. If publish-service succeeds but the storefront deploy step later fails, the tenant re-runs the storefront deploy at the same commit — composer is already correctly pointed. If the storefront deploys but CDN upload fails, publish-service rejects the call and composer keeps showing the prior tenant version; storefront is running new code composer can't edit for. This is the least-bad failure mode and is eventually consistent once the tenant CI retries the CDN step.

9. Tenant template drift management

Problem this addresses. Every tenant repo starts from a template. Six months in, the template evolves (new lint rule, new federation config, new env var). Tenants have modified files the template also modifies. Merging updates is a real problem, distinct from bumping npm dependencies.

Design

  1. The tenant template is a Copier template (or Cruft, if the team prefers Python-free tooling — pick one, use everywhere).
  2. Each generated tenant repo has a .copier-answers.yml committed at generation, recording the template version.
  3. A scheduled workflow (weekly) runs copier update against every tenant repo, opening a PR with the diff. The tenant's own CI tests the PR; the tenant merges when convenient. Conflicts are resolved by the tenant, not by the platform team.
  4. The platform team maintains a compatibility matrix: template versions < X are considered unsupported. Tenants below the floor get a nudge from the platform team; the floor moves forward roughly quarterly.
  5. Files the template owns exclusively (federation config, tenant CI config re-export) are marked in copier.yml as keep_upstream: true — the update always wins, and the tenant is expected not to edit these.

10. Federation remote as a trust boundary

Problem this addresses. Composer loads and executes JS from tenant_registry.remoteEntryUrl. Whoever can write that field, or compromise the CDN path, gets code execution inside composer with cross-tenant edit access. This must be treated as a real trust boundary, not a "consider adding SRI later" note.

Design

  1. Tenant CI computes an SRI hash (sha384) over the built remoteEntry.js in a trusted GitHub Actions environment.
  2. The hash is sent to publish-service as part of the publish request (§8) and stored in tenant_registry.remoteEntryIntegrity.
  3. Composer's federation loader fetches the remote and verifies its content matches the stored SRI hash before evaluating. Mismatch → refuse to load, alert, show the editor a "this tenant's build is untrusted" error state.
  4. CDN bucket write permissions are scoped per-tenant path via IAM: tenant-A's CI can only write to /tenants/acme/. This prevents a compromised tenant from replacing another tenant's remoteEntry.js.
  5. Composer runs each tenant's loaded module inside an isolated realm where practical, though realistically the primary defense is the SRI check — realm isolation in the browser has limits.
// glue-core/src/composer-registry.ts
export async function getTenantComponentRegistryForComposer(tenantSlug: string) {
  const tenant = await getTenantFromRegistry(tenantSlug);

  const resp = await fetch(tenant.remoteEntryUrl, { integrity: tenant.remoteEntryIntegrity });
  if (!resp.ok) throw new UntrustedRemoteError(tenantSlug);
  // ... proceed with @module-federation/runtime init + loadRemote
}

11. Composer style sandboxing

Problem this addresses. Tenant components declare CSS variables on :root. If composer renders tenant components inline, the tenant's global styles leak into composer's own chrome (its sidebar, toolbar, modals get tenant-colored).

Design

  1. Composer renders tenant components inside an iframe with a fresh document. The iframe hosts its own React tree that consumes the loaded remote. This gives real style isolation, plus a natural boundary if a tenant component crashes.
  2. Communication between composer's outer app (drag/drop, prop panel) and the iframe (rendered preview) goes over postMessage with a typed protocol defined in glue-types.
  3. The iframe URL is same-origin to composer for cookie/auth reasons; the isolation is document-level, not origin-level.

12. Shared types & type flow

Problem this addresses. Types like PageSection, Cart, Order, TenantRow, and the composer↔iframe message protocol appeared implicitly in prior versions. Without one owning package they drift.

Design

Introduce glue-types as the base package everything depends on. It contains only type declarations and shared JSON Schema definitions — no runtime code. Every other package peer-depends on it. Bumping glue-types is the only way a shared shape changes; this makes drift structurally visible in package.jsons.

15. glue-data — data access layer

// src/db.ts — tenant-scoped by construction
import type { TenantRow } from '@clouda-inc/glue-types';

export async function getTenantDb(tenantSlug: string) {
  const tenant = await getTenantFromRegistry(tenantSlug);
  if (tenant.isolationMode === 'db-per-tenant') {
    const conn = await getConnection(tenant.dbConnectionRef);
    return {
      find:   (c: string, q: object)             => conn.collection(c).find(q).toArray(),
      insert: (c: string, doc: object)           => conn.collection(c).insertOne(doc),
      update: (c: string, q: object, p: object)  => conn.collection(c).updateOne(q, { $set: p }),
    };
  }
  const conn = await getSharedConnection();
  return {
    find:   (c, q)    => conn.collection(c).find({ ...q, tenantId: tenant.slug }).toArray(),
    insert: (c, doc)  => conn.collection(c).insertOne({ ...doc, tenantId: tenant.slug }),
    update: (c, q, p) => conn.collection(c).updateOne({ ...q, tenantId: tenant.slug }, { $set: p }),
  };
}

// src/tenant-registry.ts — cursor-paginated listing
export async function listTenants(cursor?: string, limit = 50) {
  const conn = await getRegistryConnection();
  const q = cursor ? { slug: { $gt: cursor } } : {};
  const rows = await conn.collection('tenant_registry').find(q).sort({ slug: 1 }).limit(limit).toArray();
  return { rows, nextCursor: rows.length === limit ? rows[rows.length - 1].slug : null };
}
DB isolation choice is not cheaply reversible. Migrating from shared-DB-with-discriminator to DB-per-tenant later is enormously expensive once real data is in shared collections. Default recommendation: DB-per-tenant. Only choose shared-DB if you have a specific reason and low tenant count where the operational cost of many DBs matters more than isolation.

const ajv = new Ajv({ removeAdditional: 'failing' });

export async function publishPage(tenantSlug: string, page: PageDraft, registry: ComponentRegistry) {
  for (const section of page.sections) {
    const schema = registry[section.componentType]?.schema;
    if (schema && !ajv.validate(schema, section.props)) {
      throw new ValidationError(section.componentType, ajv.errors);
    }
  }
  const db = await getTenantDb(tenantSlug);
  return db.update('pages', { path: page.path }, page);
}

// src/composer-registry.ts — SRI-verified federation loader (composer-only)
import { init, loadRemote } from '@module-federation/runtime';
import { getTenantFromRegistry } from '@clouda-inc/glue-data';

const initialized = new Set<string>();

export async function getTenantComponentRegistryForComposer(tenantSlug: string) {
  const tenant = await getTenantFromRegistry(tenantSlug);

  // SRI verification before any code is executed
  const resp = await fetch(tenant.remoteEntryUrl, { integrity: tenant.remoteEntryIntegrity });
  if (!resp.ok) throw new UntrustedRemoteError(tenantSlug);

  const remoteName = `tenant_${tenantSlug}`;
  if (!initialized.has(remoteName)) {
    init({ name: 'glue_composer_host', remotes: [{ name: remoteName, entry: tenant.remoteEntryUrl }] });
    initialized.add(remoteName);
  }
  const mod = await loadRemote<{ default: any }>(`${remoteName}/components`);
  return mod.default;
}

20. publish-service — deployment coordinator

A small HTTP service (Node/Fastify or similar) that owns all writes to tenant_registry. Deployed once. See §8 for its API contract. Its only responsibilities:

  1. Verify per-tenant JWT auth against public keys in tenant_registry.
  2. HEAD-check the remoteEntry URL exists and re-verify SRI hash.
  3. Idempotent atomic write of the tenant row.
  4. Emit a webhook to composer + the tenant's storefront on successful update, so caches invalidate eagerly.

Keeping this as a separate service — rather than letting CI write the registry directly — is what makes point 3 of §4 (deploy atomicity), points 1–4 of §8 (auth, verification, idempotency, atomicity), and §10 (SRI enforcement) all enforceable in one place. It is small but architecturally load-bearing.

21. Build phases — order of development & integration points

This section is the sequencing guide. Build in this order so that each phase produces something integratable and testable before the next begins. Skipping ahead — or building repos in isolation and integrating at the end — is what causes multi-repo systems to melt down at integration time.

Phase 1 — Type foundation (1–2 days)

Build: glue-types repo with initial types (PageSection, TenantRow, Cart/Order, ComponentRegistry, composer-preview protocol). Publish 0.1.0 to CodeArtifact.

Why first: everything else imports these. Getting them wrong forces re-work across every downstream repo.

Done when: package publishes, has a README documenting each exported type, and downstream repos can install it.

Phase 2 — Data layer (2–3 days)

Build: glue-data with getTenantDb, getTenantFromRegistry, the LRU cache with stale-if-error, and a working tenant_registry DB schema. Ship both isolation modes behind the same interface.

Integration point: integration tests boot against a real (dockerised) database and assert that queries against a missing tenant fail closed. Cache invalidation is unit-tested against a mock DB.

Done when: a test can insert a tenant row, query for it, invalidate, and re-read.

Phase 3 — Base UI + component contract (3–5 days)

Build: glue-components with 5–10 base components, each with schema.json. Include the CI meta-schema validator. Include theme.css and theme.json.

Integration point: writing a downstream test app that imports glue-components, renders a Button, and applies a theme override to verify the CSS variable pipeline works.

Phase 4 — Framework package (3–5 days)

Build: glue-core with SectionRenderer, resolveTheme, and page-data hooks. Defer the composer-registry / federation loader piece to Phase 8 (composer needs it, tenant apps do not).

Integration point: a throwaway test app that imports glue-core, passes it a hand-built registry, and renders a fixed page. This proves the tenant-app pattern before any tenant exists.

Phase 5 — Service layer (2–4 days)

Build: glue-services with createCheckoutHandler, publishPage (with Ajv validation), verifyComposerHmac. Depends on glue-data (Phase 2).

Integration point: integration test where a page draft with a schema violation is rejected and one that passes writes to the DB successfully.

Phase 6 — publish-service (2–3 days)

Build: the small HTTP service specified in §20, with JWT auth, idempotency, SRI verification against a real CDN bucket, and atomic registry writes. Deploy a staging instance.

Integration point: a scripted end-to-end test that uploads a fake remoteEntry.js to CDN, POSTs a publish request, verifies the row is written, and rejects a second call with the same idempotency key.

Phase 7 — Tenant template (5–7 days)

Build: the Copier/Cruft template producing a tenant repo, including: schemas-to-types generator, ESLint config with the federation-reexport rule, tenant-CI-config, prod build scripts, the invalidate route, sample components (Hero, ProductCard), federation config.

Integration point: generate one throwaway tenant repo from the template, run npm run dev, watch a fully working storefront render locally against a seeded database. This is the first true end-to-end integration of everything built so far.

Critical: do not proceed past this phase until npm run dev gives instant HMR on a component change. If it doesn't, something is wrong that the rest of the plan will amplify.

Phase 8 — Composer (7–10 days)

Build: add the composer-registry loader to glue-core (this was deferred from Phase 4). Build glue-composer: tenant picker, iframe-sandboxed preview, drag-drop page builder, schema-driven prop editor, save flow calling publishPage.

Integration point: deploy the Phase 7 tenant's remoteEntry.js to CDN, call publish-service, open composer, load the tenant, edit a page, save, refresh the storefront and see the change. This is the second full-system integration test.

Phase 9 — Second tenant + drift check (2–3 days)

Build: generate a second tenant repo from the template. Prove that composer can pick between the two, that changing one has no effect on the other, and that a template update produces a working PR against both.

Why this phase: problems in the multi-tenant architecture only appear once N>1. Doing this before scale-up is much cheaper than after.

Phase 10 — Hardening (ongoing)

Observability (§26), the bump-all mechanism for glue-core (§24), operational runbooks for the failure modes in §6/§8/§10, security review of the federation trust boundary. This phase never fully closes — it is the ongoing platform work after the system is live.

22. Local dev workflows

Tenant developer, day-to-day

npm run dev in the tenant repo. Full HMR against local components and services. The prebuild hook regenerates types from schemas so a schema edit reflects instantly in TypeScript autocomplete.

Tenant developer, previewing in composer

npm run dev:expose serves the composer-facing remoteEntry.js at localhost:3001. In composer, opening a URL like ?tenant=acme&remoteOverride=http://localhost:3001/_next/static/chunks/remoteEntry.js uses that URL for the session only, without touching tenant_registry. Session-scoped, disappears on close.

Platform developer, working on a shared package

Use npm link or yarn link against a checked-out tenant repo. Once the change is stable, publish a pre-release version (2.4.0-next.1), bump the tenant's package.json, and confirm before publishing a stable version.

23. CI/CD pipelines end-to-end

Shared packages (glue-types, -components, -core, -data, -services)

  1. Lint, type-check, test.
  2. Semver-check against the previously published version — any breaking change without a major bump fails.
  3. Publish on merge to main.
  4. Trigger the bump-all workflow to open PRs against every tenant repo.

Tenant repo (single atomic pipeline)

  1. Install, lint (including the federation-reexport rule), generate types from schemas.
  2. Type-check, unit-test.
  3. Two parallel build outputs from the same commit SHA:
    • next build → the storefront (deployed to the tenant's hosting).
    • next build --federation-only → remoteEntry.js (uploaded to cdn.clouda.io/tenants/<slug>/<sha>/).
  4. Compute SRI hash of the uploaded remoteEntry.js.
  5. Deploy the storefront to hosting with BUILD_COMMIT=<sha> injected.
  6. Call publish-service with a JWT signed with the tenant's repo key, passing SHA, version, remote URL, SRI hash, idempotency key. Retry with the same idempotency key on transient failure.
  7. On success, publish-service updates tenant_registry and webhooks composer + storefront to invalidate caches.

glue-composer

Standard build/deploy. Not touched during tenant work. Redeploys are triggered by composer feature work only.

Bump-all mechanism

A scheduled GitHub Actions workflow runs against every tenant repo on publish of any pinned shared package. It opens a PR bumping the pin; the tenant's own CI runs; the tenant merges when convenient. Combined with the template update workflow (§9), most tenants stay within one minor version of the platform head without any manual intervention.

26. Observability & failure UX

Tags on every log/trace

tenantSlug, glueCoreVersion, deployedCommit. This makes per-tenant issue filtering trivial and lets you spot version-adoption patterns.

Metrics to dashboard

  • Composer: remote load success rate, per-tenant. Alert on a specific tenant's rate dropping.
  • Storefront: tenant_registry cache hit rate, stale-if-error events, invalidate-webhook receipts. Alert on stale-if-error being frequent.
  • publish-service: publish success rate, SRI-mismatch rejections (any nonzero is a security event to review), rollback frequency.
  • Platform-wide: glueCoreVersion adoption histogram — spot tenants stuck on old versions.

Failure UX in composer

A tenant's remote failing to load should not gray-screen composer. It should show the editor: "Tenant Acme's build v42 (SHA 9a7f2c1b) can't be loaded. Recent versions: [v41, v40, v39]. Roll back?" — with a button that calls publish-service's rollback endpoint. This turns a broken deploy from a support ticket into a self-service recovery.

27. Tenant onboarding checklist

  1. Generate the tenant repo from the current template (Copier); commit the .copier-answers.yml.
  2. Set TENANT_SLUG and TENANT_FEDERATION_NAME.
  3. Provision the tenant's database (§15's isolation mode); record dbConnectionRef.
  4. Generate the tenant's JWT signing keypair; store the private key in CI secrets; register the public key against the tenant row.
  5. Insert the initial tenant_registry row (slug, domains, isolationMode, glueCoreVersion — remoteEntryUrl left null until first deploy).
  6. Wire the tenant CI with the pipeline described in §23.
  7. Create the tenant's GitHub team scoped to this one repo; add their developers.
  8. Trigger the first deploy; verify publish-service accepts it and composer can load the tenant.
  9. Hand off the tenant developer onboarding docs (theme tokens, component patterns, schema evolution rules).

Glossary

  • Deploy atomicity (§4): the property that the storefront's compiled code and the composer's remote view of it are always built from the same commit and made visible together.
  • Additive-only schemas (§5): the rule that schema.json fields may be added or deprecated but never renamed or removed.
  • Stale-if-error (§6): the fallback behavior where the storefront serves the last good tenant_registry snapshot if the registry is unreachable.
  • Remote / host (Module Federation): a tenant repo is a remote (exposes components); glue-composer is the only host. The storefront is neither.
  • publish-service (§8, §20): the small service that owns all writes to tenant_registry, enforcing auth, verification, idempotency, and SRI checks in one place.

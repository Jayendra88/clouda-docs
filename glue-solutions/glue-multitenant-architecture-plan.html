Every section includes concrete file layouts and working code patterns, not just prose.

1. Goals & constraints

  • Tenants can edit only their own components and services — never the shared component library, core rendering engine, or other tenants' code.
  • Running npm run dev inside a tenant's repo makes the storefront (glue-main) render live using that tenant's components — the same experience as VTEX FastStore's local linking.
  • Tenants can consume prebuilt components from glue-components as-is, or wrap/restyle them without forking.
  • Tenants can override the site's visual theme independently of code changes (and non-developers can do it from the composer).
  • glue-composer runs as one standalone deployment and resolves each tenant's components dynamically at request time — it never bundles tenant code at build time (VTEX Headless CMS model).
  • glue-main and glue-composer ship once and are barely touched after initial development; all day-to-day tenant work happens in the tenant repo.

2. Repository topology

┌────────────────────────────────────────────────────────────┐ │ PUBLISHED PACKAGES (npm / AWS CodeArtifact) │ │ │ │ glue-components — base component library + schema.jsons │ │ glue-core — section renderer, layout, data hooks │ │ glue-data — tenant-scoped data access layer │ │ glue-services — generic service layer + hook contract│ └────────────────────────────────────────────────────────────┘ ▲ consumed as npm deps ▲ dynamic remote (Module Federation) │ │ ┌────────────────┴───────────────┐ ┌──────────────┴───────────────┐ │ glue-main (deployed once) │ │ glue-composer (deployed once)│ │ storefront renderer │ │ standalone CMS │ └────────────────▲───────────────┘ └──────────────▲───────────────┘ │ resolves tenant's remoteEntry.js at runtime, per request │ (URL comes from tenant_registry) ┌────────────────┴──────────────────────────────────────────────────┐ │ tenant-repo-acme (one repo per tenant, tenant-owned) │ │ components/ — tenant components + overrides of glue-components │ │ services/ — tenant-specific API routes / business logic │ │ theme/ — tenant design tokens │ │ exposes a Module Federation remote: remoteEntry.js │ └─────────────────────────────────────────────────────────────────────┘

The key change from a build-time-linked storefront to this model: glue-main no longer imports tenant components at build time. It imports glue-core (the rendering engine) as a normal npm dependency, and resolves the tenant's actual component set as a Module Federation remote, at request time, based on which tenant is being served. This is what makes a tenant's local npm run dev instantly visible in glue-main without anyone rebuilding glue-main.

3. Repo-by-repo responsibilities

glue-components

  • Prebuilt, business-agnostic components (Button, Hero, ProductCard, Carousel, …), each with a co-located schema.json describing its editable props.
  • Ships a default theme.css defining every CSS variable a component reads. Tenants override values, never the variable names.
  • Published to CodeArtifact/npm. Tenants install it as a normal dependency and either use components directly or wrap them.

glue-core

  • SectionRenderer — walks a page's section list and renders the matching component from whichever registry (base + tenant) is resolved for the current request.
  • getTenantComponentRegistry(tenantSlug) — the Module Federation resolution logic (detailed in §5–6).
  • Layout primitives (grid, container, spacing) shared by every tenant's storefront.

glue-main

  • The one deployed Next.js app that renders every tenant's live storefront (host header / subdomain decides which tenant).
  • Depends on glue-core, glue-data as npm packages. Contains no tenant business logic.
  • Barely changes after initial build — new tenant capability comes from the tenant repo, not from touching this app.

glue-composer

  • Standalone CMS/editor. Same dynamic remote resolution as glue-main, but also reads each component's schema.json to build the property editor and page-builder UI.
  • Owns page/content CRUD via glue-services; has no compiled dependency on any tenant's code.

glue-data / glue-services

  • glue-data: tenant-scoped DB access, the tenant_registry reader, page/content repositories.
  • glue-services: generic business logic (checkout, page publish workflow, content validation against schema.json) exposed as functions plus a hook contract tenants can plug into from their own repo — see §10.

tenant-repo-<slug>

One per tenant, tenant-owned, covered fully in the next section.

4. Tenant repo — deep dive

tenant-repo-acme/ ├── src/ │ ├── components/ │ │ ├── Hero/ │ │ │ ├── index.tsx ← tenant-authored component │ │ │ └── schema.json ← editable-props contract for composer │ │ ├── ProductCard/ │ │ │ ├── index.tsx ← wraps a glue-components component │ │ │ └── schema.json │ │ └── index.ts ← registry export (see §8) │ ├── services/ │ │ ├── pricing.ts ← tenant-specific business logic │ │ └── api/ │ │ └── checkout/route.ts ← extends glue-services checkout handler │ ├── theme/ │ │ ├── tokens.json ← design tokens (colors, type, spacing) │ │ └── theme.css ← generated from tokens.json at build │ └── federation/ │ ├── expose-components.ts ← Module Federation "exposes" entry │ └── expose-theme.ts ├── next.config.js ← ModuleFederationPlugin config ├── tenant.config.json ← slug, DB ref, feature flags ├── .env.local ← service configs, secrets └── package.json

Everything the tenant is allowed to touch lives inside this one repo: their components, their theme, and their tenant-specific service/API logic. Nothing here is imported by glue-main or glue-composer at build time — it is only ever pulled in dynamically at runtime as a Module Federation remote.

5. Module Federation: dynamic runtime remotes

We use @module-federation/enhanced (Module Federation 2.0's bundler-agnostic runtime) rather than build-time federation, because the remote URL is only known per-request (it depends on which tenant is being served, and — in local dev — points at localhost).

Tenant repo — exposing components (next.config.js)

const { ModuleFederationPlugin } = require('@module-federation/enhanced/webpack');

module.exports = {
  webpack(config) {
    config.plugins.push(
      new ModuleFederationPlugin({
        name: process.env.TENANT_FEDERATION_NAME, // e.g. "tenant_acme"
        filename: 'static/chunks/remoteEntry.js',
        exposes: {
          './components': './src/federation/expose-components.ts',
          './theme': './src/federation/expose-theme.ts',
        },
        shared: {
          react: { singleton: true, requiredVersion: false },
          'react-dom': { singleton: true, requiredVersion: false },
        },
      })
    );
    return config;
  },
};

Tenant repo — the registry entry point (src/federation/expose-components.ts)

import Hero from '../components/Hero';
import heroSchema from '../components/Hero/schema.json';
import ProductCard from '../components/ProductCard';
import productCardSchema from '../components/ProductCard/schema.json';
import { Button } from '@clouda-inc/glue-components'; // used unmodified

const registry = {
  Hero:        { component: Hero,        schema: heroSchema },
  ProductCard: { component: ProductCard, schema: productCardSchema },
  Button:      { component: Button,      schema: null }, // no overrides needed
};

export default registry;

glue-main / glue-composer — resolving a remote at request time (packages/glue-core/src/registry.ts)

import { init, loadRemote } from '@module-federation/runtime';
import { getTenantFromRegistry } from '@clouda-inc/glue-data';

const initializedRemotes = new Set<string>();

export async function getTenantComponentRegistry(tenantSlug: string) {
  const tenant = await getTenantFromRegistry(tenantSlug);
  const remoteName = `tenant_${tenantSlug}`;

  // Local dev override takes priority — see §7
  const entryUrl = await resolveLocalOverride(tenantSlug) ?? tenant.remoteEntryUrl;

  if (!initializedRemotes.has(remoteName)) {
    init({
      name: 'glue_host',
      remotes: [{ name: remoteName, entry: entryUrl }],
    });
    initializedRemotes.add(remoteName);
  }

  const mod = await loadRemote<{ default: Record<string, any> }>(
    `${remoteName}/components`
  );
  return mod.default;
}
Cache getTenantFromRegistry results for a short TTL (e.g. 30–60s) so a new tenant deploy is picked up quickly without hitting the DB on every request.

6. Tenant registry & resolution

tenant_registry remains the single source of truth mapping a tenant slug to where its code and config live.

// tenant_registry document shape
{
  "slug": "acme",
  "remoteEntryUrl": "https://cdn.clouda.io/tenants/acme/v14/remoteEntry.js",
  "manifestUrl": "https://cdn.clouda.io/tenants/acme/v14/manifest.json",
  "version": "v14",
  "dbConnectionRef": "acme-prod-cluster",      // or a discriminator if shared-DB
  "themeOverrides": { "--color-primary": "#123456" }, // composer-managed, see §9
  "domains": ["acme.com", "acme.clouda.io"],
  "createdAt": "2026-05-01T00:00:00Z",
  "updatedAt": "2026-07-10T00:00:00Z"
}

Resolution order for which tenant is being served: incoming request's Host header → domains lookup in tenant_registry → slug. Resolve this in middleware in both glue-main and glue-composer and attach it to the request context so downstream code never re-derives it.

7. Local dev workflow (npm run dev)

The goal: a tenant developer runs one command in their own repo and sees glue-main render live with their local, hot-reloading components — no publish step, no touching glue-main.

tenant-repo package.json

{
  "scripts": {
    "dev": "clouda-dev tenant-dev"
  }
}

What clouda-dev tenant-dev does

  1. Starts next dev in the tenant repo (port 3001), which serves a live, HMR-enabled remoteEntry.js at http://localhost:3001/_next/static/chunks/remoteEntry.js.
  2. Writes a local override file .clouda/local-registry.json: 
    { "slug": "acme", "remoteEntryUrl": "http://localhost:3001/_next/static/chunks/remoteEntry.js" }
  3. Starts (or attaches to an already-running) local glue-main instance with LOCAL_TENANT_OVERRIDE=true.
  4. Opens the browser to glue-main's local URL for that tenant. From this point every save in the tenant repo hot-reloads inside glue-main.

resolveLocalOverride implementation

import fs from 'node:fs/promises';

export async function resolveLocalOverride(tenantSlug: string) {
  if (process.env.LOCAL_TENANT_OVERRIDE !== 'true') return null;
  try {
    const raw = await fs.readFile('.clouda/local-registry.json', 'utf-8');
    const override = JSON.parse(raw);
    return override.slug === tenantSlug ? override.remoteEntryUrl : null;
  } catch {
    return null;
  }
}

This is the direct equivalent of VTEX Toolbelt's local linking — the difference is it is powered by a real-time federation remote instead of a proxy tunnel.

Tenants have three ways to use the library, all valid:

Use as-isNo visual/behavior change neededexport { Button } from '@clouda-inc/glue-components'
Restyle via CSS varsSame behavior, different lookOverride --color-primary in tenant theme.css; component unchanged
Wrap/extendExtra markup or logic on top of a base componentSee below
// tenant-repo/src/components/ProductCard/index.tsx
import { ProductCard as BaseProductCard } from '@clouda-inc/glue-components';

export default function ProductCard(props) {
  return (
    <div className="acme-product-card-wrapper">
      <BaseProductCard {...props} />
      {props.isNew && <span className="acme-badge">New</span>}
    </div>
  );
}

Every tenant component — new or wrapped — needs a schema.json next to it, in the same shape glue-components uses, so the composer can build its editor without special-casing tenant components.

    "backgroundImage": { "type": "string", "title": "Background image", "format": "image" }
  },
  "required": ["headline"]
}

9. Theming system

Three layers, lowest to highest priority:

  1. Base defaults — theme.css shipped inside glue-components, defines every variable name.
  2. Tenant repo defaults — src/theme/tokens.json in the tenant repo, exposed via the ./theme Module Federation entry, compiled at tenant build time.
  3. Composer runtime overrides — non-developers tweak colors/fonts in the composer UI; these save to tenant_registry.themeOverrides and win over the tenant repo's own defaults, so a marketer can adjust the theme without a deploy.
// glue-main: theme resolution, run once per request
export async function resolveTheme(tenantSlug: string) {
  const tenant = await getTenantFromRegistry(tenantSlug);
  const registryMod = await loadRemote<{ default: Record<string, string> }>(
    `tenant_${tenantSlug}/theme`
  );
  return {
    ...registryMod.default,        // tenant repo defaults
    ...tenant.themeOverrides,      // composer-managed overrides win
  };
}

<!-- injected server-side into the <head>, scoped by data-tenant -->
<html data-tenant="acme">
  <style>
    [data-tenant="acme"] {
      --color-primary: #123456;
      --font-heading: 'Acme Sans', sans-serif;
    }
  </style>

10. Tenant-specific service layer & API routes

glue-services exposes generic business logic as a factory that accepts hooks — tenant code supplies the hooks, never forks the handler.

// packages/glue-services/src/checkout.ts
export function createCheckoutHandler(hooks: {
  beforePriceCalculation?: (cart: Cart) => Promise<Cart>;
  afterOrderCreated?: (order: Order) => Promise<void>;
} = {}) {
  return async function handler(req: Request) {
    let cart = await parseCart(req);
    if (hooks.beforePriceCalculation) cart = await hooks.beforePriceCalculation(cart);
    const order = await createOrder(cart);
    if (hooks.afterOrderCreated) await hooks.afterOrderCreated(order);
    return Response.json(order);
  };
}

// tenant-repo/src/services/api/checkout/route.ts
import { createCheckoutHandler } from '@clouda-inc/glue-services';
import { applyLoyaltyDiscount } from '../../pricing';

export const POST = createCheckoutHandler({
  beforePriceCalculation: applyLoyaltyDiscount,
});

Note this API route lives in the tenant repo and is deployed alongside the tenant's own infrastructure (or as serverless functions the tenant repo owns) — glue-main and glue-composer never proxy or host it.

11. Data access layer & multi-tenant DB

glue-data gives every consumer the same interface regardless of isolation strategy, so the strategy can change later without touching call sites.

// packages/glue-data/src/db.ts
type TenantDb = { find: (col: string, q: object) => Promise<any[]>; /* ...insert, update */ };

export async function getTenantDb(tenantSlug: string): Promise<TenantDb> {
  const tenant = await getTenantFromRegistry(tenantSlug);

  if (tenant.isolationMode === 'db-per-tenant') {
    const conn = await getConnection(tenant.dbConnectionRef); // tenant's own DB
    return { find: (col, q) => conn.collection(col).find(q).toArray() };
  }

  // shared-db-with-discriminator: tenantId is injected, never optional
  const conn = await getSharedConnection();
  return {
    find: (col, q) => conn.collection(col).find({ ...q, tenantId: tenant.slug }).toArray(),
  };
}
Recommendation carried over from the base architecture: prefer DB-per-tenant given this is a commerce platform with real per-tenant compliance/data needs. Keep the interface above either way so the decision is reversible.

12. Composer app — dynamic CMS behavior

Composer reuses the exact same getTenantComponentRegistry from glue-core — it is not a special case. It additionally reads each entry's schema to render an editor.

// glue-composer: component picker panel
async function ComponentPicker({ tenantSlug }: { tenantSlug: string }) {
  const registry = await getTenantComponentRegistry(tenantSlug);
  return Object.entries(registry).map(([name, { schema }]) => (
    <DraggableComponentTile key={name} name={name} schema={schema} />
  ));
}

// schema-driven prop editor — works for base AND tenant components identically
function PropEditor({ schema, values, onChange }) {
  return Object.entries(schema.properties).map(([field, def]: any) => (
    <FieldInput
      key={field}
      label={def.title}
      type={def.type}
      format={def.format}
      value={values[field]}
      onChange={(v) => onChange(field, v)}
    />
  ));
}

Saving a page writes a section list to the page document via glue-services' page-publish workflow, validating each section's props against that component's schema.json before persisting — this is what stops a bad edit in composer from breaking the live storefront render.

13. Repo permissions & access control

RepoTenant dev accessEnforcement
tenant-repo-<slug>Full read/writeGitHub team scoped to this one repo
glue-components, glue-core, glue-data, glue-servicesRead-only (or no access)CODEOWNERS + branch protection; only the platform team can merge
glue-main, glue-composerNo accessNot part of tenant org team at all
  • Each tenant repo's CI pipeline gets a scoped deploy token that can only publish to that tenant's own CDN path (/tenants/<slug>/...) and update only its own row in tenant_registry (via an internal API, not direct DB access).
  • Composer's theme/content editing writes go through glue-services, which enforces tenant-scoping server-side regardless of what the client sends — never trust a tenant slug coming from the request body alone; derive it from the authenticated session/host.

14. CI/CD & deployment pipelines

Tenant repo pipeline

  1. Lint + test + next build (produces the versioned remoteEntry.js + static chunks).
  2. Upload build output to https://cdn.clouda.io/tenants/<slug>/<version>/ (keep prior versions for instant rollback).
  3. Call the internal "publish tenant version" API to update tenant_registry.remoteEntryUrl and version.
  4. No redeploy of glue-main/glue-composer — the new remote is picked up on next request once the tenant_registry cache TTL expires.

glue-main / glue-composer pipeline

Standard Next.js build/deploy (e.g. AWS Amplify). Redeploys only happen for core rendering engine changes, not tenant work — this is what keeps them "barely touched."

15. End-to-end request flow

  1. Request hits glue-main at acme.com/product/123.
  2. Middleware resolves tenantSlug = "acme" from the Host header via tenant_registry.domains.
  3. getTenantComponentRegistry("acme") resolves the remote (local override if present, else tenant.remoteEntryUrl) and loads the component registry.
  4. resolveTheme("acme") merges base defaults, tenant repo tokens, and composer overrides into CSS variables injected in <head>.
  5. getTenantDb("acme") fetches the page's section list for /product/123.
  6. SectionRenderer (from glue-core) walks the sections, rendering each from the resolved registry.
  7. Response streams to the browser — a page composed entirely of the tenant's own components, running inside the shared glue-main shell.

Glossary

  • Remote / host (Module Federation) — the tenant repo is the remote (exposes code); glue-main and glue-composer are hosts (consume it at runtime).
  • tenant_registry — the routing source of truth: slug → remote URL, DB reference, theme overrides, domains.
  • schema.json — the JSON Schema contract each component publishes so the composer can build an editor for it without special-casing.

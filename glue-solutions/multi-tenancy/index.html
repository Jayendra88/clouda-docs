Multi-tenancy for the Clouda Glue platform

How to serve many tenant websites from one codebase: a tenant slug in the database, one shared Glue Composer, per-tenant deployments of glue-main, per-tenant theming, shared workspace packages, and component-level configuration. This site documents the whole approach — architecture, code, and rollout.

Proposed — awaiting approval. Nothing implemented yet.
How to read this. Start here for the big picture, then follow the sidebar top-to-bottom: each page covers one area with diagrams, code, and the concrete files it touches. The Roadmap & Risks page ties it into a phased, MVP-first plan.

The system at a glance

One monorepo builds everything. glue-main is deployed once per tenant (its own domain + a TENANT_SLUG), while a single glue-composer manages content for all tenants. Both read from one MongoDB where every document carries a tenant slug, so tenants stay isolated.

Four decisions that shape everything

Data isolation

Shared DB + tenant slug. One database; every document tagged with a tenantId, enforced by a mandatory scoping layer. A dbName field per tenant is the escape hatch to a dedicated DB later.

Deployment

Deploy glue-main per tenant. One Amplify app per tenant with a TENANT_SLUG env var and its own domain; a single shared Amplify app for glue-composer.

Theming

CSS variables + per-tenant theme files. Refactor components off hardcoded values onto tokens; load the tenant's theme file based on TENANT_SLUG.

Rollout

MVP first. Build the foundation and take one real second tenant fully live before the heavier admin/registry tooling.

From single-tenant to multi-tenant

Today the platform is strictly single-tenant: one MongoDB (webpages + webpage_content, no tenant field), one hardcoded domain, one shared component library with hardcoded brand colours, and one global role model. The proposal makes each of these tenant-aware while keeping www.clouda.io behaving exactly as it does today throughout the rollout.

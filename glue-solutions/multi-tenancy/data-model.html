Data Model & Isolation

Introduce a single tenant slug into the database and make it impossible to read or write un-scoped. Every tenant-owned document gains a tenantId, three new collections describe tenants and their people and components, tenant-leading indexes keep reads fast, and a mandatory scoping repository guarantees that no query can ever escape its tenant.

Proposed — awaiting approval. Nothing implemented yet.
The core idea. Isolation is not enforced by discipline — it is enforced by structure. A shared getTenantRepo(tenant) is the only way application code touches the database, and it forces tenantId onto every operation. An un-scoped query is not discouraged; it is unavailable.

The tenant slug

The tenant identifier is the tenant's human-readable slug — the same value passed to a glue-main deployment as TENANT_SLUG (for example clouda, acme). We store it on every tenant-owned document as tenantId. Using the slug rather than an opaque ObjectId means the value in the environment, in the URL logs, and in the database all read the same, and a scoped query needs no extra lookup to translate an env var into a key.

One value, everywhere. TENANT_SLUG (env) = tenants.slug = the tenantId stamped on webpages and webpage_content. If they ever disagree, the tenant is misconfigured — there is nothing to reconcile.

New collections

Three collections describe the multi-tenant world itself. They are the only place tenant metadata lives; the content collections just reference a slug.

packages/db · collection shapests
// tenants — one document per tenant; slug is the tenantId used everywhere else
interface Tenant {
  _id: ObjectId;
  slug: string;              // UNIQUE — e.g. "clouda"; equals TENANT_SLUG
  name: string;
  domains: string[];         // all hostnames that map to this tenant
  primaryDomain: string;
  status: "active" | "suspended" | "archived";
  dbName: string;            // "glue" today; reserved for DB-per-tenant later
  baseUrl: string;
  theme: Record<string, unknown>;      // tokens / theme-file ref (see Theming)
  analytics: { gtmId?: string };
  integrations: Record<string, unknown>;  // Zoho, etc.
}

// tenant_users — which user belongs to which tenant, with which roles
interface TenantUser {
  _id: ObjectId;
  userId: string;            // Cognito subject
  tenantId: string;          // tenant slug
  roles: ("admin" | "editor" | "viewer")[];
  status: "active" | "invited" | "disabled";
  // UNIQUE { userId, tenantId }
}

// component_registry — which components exist and who may use them
interface ComponentRegistryEntry {
  _id: ObjectId;
  name: string;              // UNIQUE
  category: string;
  visibility: "global" | "tenant-list" | "tenant-specific";
  tenantIds: string[];       // used when visibility is tenant-list / tenant-specific
  status: "active" | "deprecated";
}
tenants owns users and pages; each page has exactly one content document; the registry references tenants by slug.

tenantId on existing collections

The two content collections each gain a tenantId holding the tenant slug. Nothing is removed — it is a purely additive field.

webpages

Existing shape {_id, name, type, status, timestamps} gains tenantId. The page list in Composer becomes a scoped query.

webpage_content

Existing shape {_id, parentId, name, status, blocks[], settings[], pagePath, timestamps} gains tenantId, denormalised from its parent.

The denormalisation matters. getWebpageContentBySlug is the hot read path on every public page request in glue-main. Carrying tenantId directly on webpage_content means that read filters {tenantId, pagePath, status:"live"} against a single collection — it never has to join back to webpages to discover which tenant the content belongs to.

Indexes

Every scoped query starts with tenantId, so every compound index must lead with it. That keeps each tenant's slice of a collection contiguous in the index and makes cross-tenant scans structurally unlikely. A partial-unique index guarantees one live page per path per tenant.

packages/db/scripts/indexes.jsjs
// webpage_content — hot read path leads with tenantId
db.webpage_content.createIndex({ tenantId: 1, pagePath: 1, status: 1 });
db.webpage_content.createIndex({ tenantId: 1, parentId: 1 });

// webpages — listing + lookups
db.webpages.createIndex({ tenantId: 1, type: 1, status: 1 });
db.webpages.createIndex({ tenantId: 1, status: 1 });

// one LIVE page per path, per tenant (partial-unique)
db.webpage_content.createIndex(
  { tenantId: 1, pagePath: 1 },
  { unique: true, partialFilterExpression: { status: "live" } }
);

// registry + membership uniqueness
db.tenants.createIndex({ slug: 1 }, { unique: true });
db.tenant_users.createIndex({ userId: 1, tenantId: 1 }, { unique: true });
db.component_registry.createIndex({ name: 1 }, { unique: true });

The scoping repository

A single shared factory, getTenantRepo(tenant) (it lives in packages/db — see the Shared Packages page), is the only way application code obtains a collection. It returns tenant-bound collection wrappers, never the raw driver collection. Each wrapper spreads { tenantId } last onto every read/update/delete filter (so a caller cannot override it) and forces tenantId onto every insert/replace. Because the raw collection is never handed out, an un-scoped query is not merely discouraged — it is impossible to express.

packages/db/src/repository.tsts
function scoped<T>(col: Collection<T>, tenantId: string) {
  // tenantId is spread LAST so a caller-supplied filter can never override it
  const scope = (f: Filter<T> = {}) => ({ ...f, tenantId });

  return {
    find: (f?: Filter<T>) => col.find(scope(f)),
    findOne: (f?: Filter<T>) => col.findOne(scope(f)),
    updateOne: (f: Filter<T>, u: UpdateFilter<T>) =>
      col.updateOne(scope(f), u),
    deleteOne: (f: Filter<T>) => col.deleteOne(scope(f)),

    // writes FORCE tenantId — caller cannot set a different one
    insertOne: (doc: OptionalId<T>) =>
      col.insertOne({ ...doc, tenantId }),
    replaceOne: (f: Filter<T>, doc: T) =>
      col.replaceOne(scope(f), { ...doc, tenantId }),
  };
}

export async function getTenantRepo(tenant: string) {
  const db = await startMongo();      // only packages/db may call this
  return {
    webpages: scoped(db.collection("webpages"), tenant),
    content: scoped(db.collection("webpage_content"), tenant),
  };
  // raw collections are never returned
}
CI gate. An ESLint rule (run in CI) forbids .collection("webpages") / .collection("webpage_content") and any import of startMongo outside packages/db. The repository is the single door; the lint rule keeps it the only door.

Migration & backfill

The migration is additive, idempotent, and reversible. Application code ships first, defaulting missing tenantId to "clouda", so nothing breaks before the data is touched. Indexes are built before the backfill so the write and every subsequent read land on a tenant-leading index. The backfill itself only sets the field where it is absent, so it can be re-run safely, and it is reversible by simply dropping the field.

  1. Create tenants + default tenantInsert the tenants collection and a single default document with slug: "clouda".
  2. Build tenant-leading indexesRun the index script so the backfill writes and later reads use the new compound indexes.
  3. Backfill existing documentsupdateMany only where tenantId is absent — idempotent by construction.
  4. VerifyAssert zero remaining un-tagged documents before considering the tenant live.
packages/db/scripts/backfill.jsjs
// 1 — default tenant (idempotent via upsert on unique slug)
db.tenants.updateOne(
  { slug: "clouda" },
  { $setOnInsert: { slug: "clouda", name: "Clouda", status: "active",
                    dbName: "glue", domains: ["www.clouda.io"] } },
  { upsert: true }
);

// 2 — indexes already built (see indexes.js) BEFORE this step

// 3 — backfill only where the field is absent → safe to re-run
db.webpages.updateMany(
  { tenantId: { $exists: false } },
  { $set: { tenantId: "clouda" } }
);
db.webpage_content.updateMany(
  { tenantId: { $exists: false } },
  { $set: { tenantId: "clouda" } }
);

// 4 — verify: both must be 0
print(db.webpages.countDocuments({ tenantId: { $exists: false } }));
print(db.webpage_content.countDocuments({ tenantId: { $exists: false } }));

// reversible: db.<col>.updateMany({}, { $unset: { tenantId: "" } })
Order guarantees safety. Ship code defaulting to clouda → build indexes → backfill → verify === 0. At no point does www.clouda.io behave differently, and each step can be re-run or rolled back independently.

Theming

Give each tenant its own look without forking components. Each tenant has one theme.json; from it we render tokens as CSS variables (colour, shape) on the server, and put variants and behaviour (hover style, scroll behaviour) on a React context via a ThemeProvider. Components read the context and reflect it onto the markup, so the same HeaderMain can differ in colour, shape, hover, and scroll behaviour per tenant.

Proposed — awaiting approval. Nothing implemented yet.
Companion documents. This page is the doc-site version. The multi-tenancy-plan theming deep-dive and plan.md carry the extended worked example, the full token catalogue, and the component-migration order this section is consistent with.

1CSS variables instead of hardcoded values

Components today bake in colours, borders, radii and spacing (the library is full of literal #F96F2D and fixed pixels). The first move is to route those through semantic CSS custom properties — a --glue-* token set — and map Tailwind onto the same tokens, so every existing utility class (bg-bg, text-fg, rounded) becomes theme-aware for free. Colours are stored as space-separated RGB channels so Tailwind's <alpha-value> still works (e.g. bg-surface/80).

packages/components/theme/tokens.css — the base (default Clouda) themecss
:root {
  /* colours: RGB channels so `/ <alpha-value>` works */
  --glue-color-bg: 10 10 10;
  --glue-color-fg: 245 245 245;
  --glue-color-brand: 249 111 45;      /* #F96F2D */
  --glue-color-surface: 23 26 33;
  --glue-color-surface-fg: 245 245 245;
  /* shape / type */
  --glue-radius: 14px;
  --glue-font-sans: "Inter", system-ui, sans-serif;
}
glue-components/tailwind.config.js — glue-main & glue-composer use tailwind.config.ts (export default)js
const rgb = (v) => `rgb(var(${v}) / <alpha-value>)`;

module.exports = {
  theme: { extend: {
    colors: {
      bg:        rgb("--glue-color-bg"),
      fg:        rgb("--glue-color-fg"),
      brand:     rgb("--glue-color-brand"),
      surface:   rgb("--glue-color-surface"),
      "surface-fg": rgb("--glue-color-surface-fg"),
    },
    borderRadius: { DEFAULT: "var(--glue-radius)" },
    fontFamily:   { sans: "var(--glue-font-sans)" },
  } },
};
Tokens are an additive fallback. An explicit prop always wins; the token is only the fallback — style={{ background: props.bg ?? "rgb(var(--glue-color-surface))" }}. An un-migrated component keeps behaving exactly as before, so you migrate one component at a time.

2One theme.json per tenant

Each tenant/brand has a single theme.json — the one source of truth for its look. It holds three things:

tokens

Values for the --glue-* variables (colour channels, radius, font). These become CSS variables (Section 3).

variants

Structural choices the theme selects — e.g. header shape: "pill" | "square". Delivered via context.

behaviour

Interaction flags — hover style, scroll behaviour. Also delivered via context and branched on in JS.

glue-main/src/themes/acme.jsonjson
{
  "name": "acme",
  "tokens": {
    "color-bg": "255 255 255",
    "color-fg": "17 24 39",
    "color-brand": "37 99 235",        // Acme blue
    "color-surface": "248 250 252",
    "color-surface-fg": "17 24 39",
    "radius": "4px",                       // sharper corners
    "font-sans": "Poppins, system-ui, sans-serif"
  },
  "variants":  { "header": { "shape": "pill" } },
  "behaviour": { "header": { "hover": "underline", "scroll": "solid" } }
}

The shape is typed once, in the shared package, so a theme can't request a variant or behaviour that doesn't exist:

packages/components/theme/types.tsts
export type HeaderShape  = "flat" | "pill" | "square";
export type HeaderHover  = "underline" | "fill" | "none";
export type HeaderScroll = "fade" | "solid" | "hide" | "shrink";

export interface Theme {
  name: string;
  tokens: Record<string, string>;                 // var name (no --glue-) -> value
  variants:  { header: { shape: HeaderShape } };
  behaviour: { header: { hover: HeaderHover; scroll: HeaderScroll } };
}

3Loading theme.json by tenant & injecting tokens

Because glue-main runs one deploy per tenant, the active tenant is known from TENANT_SLUG. A tiny registry maps the slug to its imported theme.json (falling back to the base theme for an unknown slug). A DB-backed loader could replace this later without changing callers.

glue-main/src/themes/index.ts — slug → Themets
import type { Theme } from "@clouda/components/theme";
import base   from "./base.json";      // default Clouda theme
import acme   from "./acme.json";
import brandb from "./brandb.json";

const registry: Record<string, Theme> = { acme, brandb };

export function getTheme(slug: string): Theme {
  return (registry[slug] ?? base) as Theme;   // unknown slug -> base defaults
}

The token half is turned into a :root { … } string and injected as a <style> in the server-rendered <head> — so the correct colours are present on the very first paint (no flash of the wrong theme):

glue-main/src/lib/theme.ts — tokens → CSS variablests
export function themeToCssVars(tokens: Record<string, string>): string {
  const body = Object.entries(tokens)
    .map(([name, value]) => `--glue-${name}: ${value};`)
    .join(" ");
  return `:root { ${body} }`;   // e.g. ":root { --glue-color-brand: 37 99 235; … }"
}

The root layout ties it together: load the tenant's theme, inject the tokens, and wrap the app in the ThemeProvider so the variants + behaviour reach every component through context.

glue-main/src/app/layout.tsxtsx
import { getTheme } from "@/themes";
import { themeToCssVars } from "@/lib/theme";
import { ThemeProvider } from "@clouda/components/theme";

export default async function RootLayout({ children }: { children: React.ReactNode }) {
  const slug  = process.env.TENANT_SLUG!;   // set per Amplify deploy
  const theme = getTheme(slug);              // this tenant's theme.json
  const vars  = themeToCssVars(theme.tokens);

  return (
    <html lang="en" data-tenant={slug}>
      <head>
        {/* tokens as CSS variables — server-rendered, no FOUC */}
        <style id="glue-theme" dangerouslySetInnerHTML={{ __html: vars }} />
      </head>
      <body>
        {/* variants + behaviour travel via React context */}
        <ThemeProvider value={theme}>{children}</ThemeProvider>
      </body>
    </html>
  );
}
Two channels, one source. theme.json is loaded once; its tokens go out as CSS (styling that CSS can express), and its variants + behaviour go into context (choices JS must read). Both come from the same file, so a tenant is configured in exactly one place.

4The ThemeProvider and useTheme hook

The provider is a small client component in the shared package: it creates a React context and exposes a useTheme() hook that returns the current tenant's Theme. It throws if used outside a provider, so a component can never silently render without a theme.

packages/components/theme/ThemeProvider.tsxtsx
"use client";
import { createContext, useContext } from "react";
import type { Theme } from "./types";

// null default lets useTheme() detect "used outside a provider"
const ThemeContext = createContext<Theme | null>(null);

export function ThemeProvider(
  { value, children }: { value: Theme; children: React.ReactNode }
) {
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}

export function useTheme(): Theme {
  const theme = useContext(ThemeContext);
  if (!theme) throw new Error("useTheme must be used within <ThemeProvider>");
  return theme;
}
Server-safe. The provider is a client component, but it only wraps children; the root layout stays a server component. Because the token CSS is injected server-side (Section 3), the base look is correct even before React hydrates — the context only drives variant/behaviour decisions, which are interactive concerns anyway.

5Components read the context & apply styles

A themed component reads its variants + behaviour from useTheme() and reflects them onto data-attributes; the CSS keys on those attributes and pulls colours from the tokens. Colour and simple shape are pure CSS (no per-tenant code); the one genuinely dynamic behaviour — scroll — comes from a hook (Section 6). This is exactly how the same HeaderMain renders differently per tenant.

packages/components/src/HeaderMain/HeaderMain.tsxtsx
"use client";
import { useTheme } from "@clouda/components/theme";
import { useScrolledPastTop } from "@clouda/components/hooks";

export function HeaderMain({ links, bg }: HeaderProps) {
  // read the tenant's choices from context
  const { variants, behaviour } = useTheme();
  const { shape } = variants.header;              // "pill" | "square" | "flat"
  const { hover, scroll } = behaviour.header;      // hover + scroll modes
  const solid = useScrolledPastTop(78);           // true once scrolled past 78px

  return (
    <header
      className="headerMain"
      data-shape={shape}
      data-hover={hover}
      data-scroll={scroll}
      data-solid={solid ? "true" : "false"}
      // explicit prop still wins; the token is only the fallback (precedence)
      style={{ background: bg ?? "rgb(var(--glue-color-surface))" }}
    >
      {links.map((l) => (
        <a key={l.href} href={l.href} className="navLink">{l.label}</a>
      ))}
    </header>
  );
}
packages/components/src/HeaderMain/HeaderMain.csscss
/* shape variant → declarative CSS */
.headerMain[data-shape="pill"] .navLink   { border-radius: 999px; }
.headerMain[data-shape="square"] .navLink { border-radius: 0; }

/* hover behaviour → declarative CSS, colours from tokens */
.headerMain[data-hover="underline"] .navLink:hover { text-decoration: underline; }
.headerMain[data-hover="fill"] .navLink:hover {
  background: rgb(var(--glue-color-brand));
  color:      rgb(var(--glue-color-surface-fg));
}

/* scroll behaviour → driven by data-solid (from the hook) + data-scroll */
.headerMain[data-scroll="solid"][data-solid="true"] {
  background: rgb(var(--glue-color-surface));
}
What this retires. The real HeaderMain hardcodes #F96F2D and guesses light-vs-dark with a brittle isLightTheame hex check plus AutoSwapColor.ts. Both go away: paired surface/surface-fg tokens carry a guaranteed-legible foreground, so no colour is inferred at runtime.

6useScrolledPastTop — the scroll hook

What it does: a small client hook that returns a boolean — false at the top of the page and true once the user has scrolled past a threshold (in pixels). It is the one dynamic input the header needs: "have we scrolled down far enough to change the header?" The HeaderMain above feeds its result into data-solid, and the CSS turns the header opaque (or, for other modes, shrinks/hides it). Keeping the scroll logic in a hook means the component stays declarative and the behaviour is reusable.

  • Starts false on the server and first paint, so SSR and hydration match (no mismatch warning).
  • Listens with a passive scroll listener throttled by requestAnimationFrame — cheap, no layout thrash.
  • Syncs once on mount (so a reload part-way down the page is correct) and cleans up on unmount.
packages/components/hooks/useScrolledPastTop.tsts
"use client";
import { useState, useEffect } from "react";

// true once the page has scrolled past `threshold` px from the top.
export function useScrolledPastTop(threshold = 0): boolean {
  const [past, setPast] = useState(false);   // false on server & first paint

  useEffect(() => {
    let ticking = false;
    const onScroll = () => {
      if (ticking) return;
      ticking = true;
      requestAnimationFrame(() => {
        setPast(window.scrollY > threshold);
        ticking = false;
      });
    };
    onScroll();                                   // sync to current position on mount
    window.addEventListener("scroll", onScroll, { passive: true });
    return () => window.removeEventListener("scroll", onScroll);
  }, [threshold]);

  return past;
}
Why a hook, not inline scroll code. The real header wires its own scroll listener and a react-headroom wrapper. Extracting "am I past the top?" into useScrolledPastTop lets several components (header, back-to-top button, sticky bars) share one throttled listener contract, and keeps the behaviour testable in isolation.

7Precedence

Everything is a fallback chain, most-general to most-specific. An explicit prop or block setting always wins; the token is only what fills in when nothing more specific was provided.

LevelSourceWins over
Tenant theme:root vars from theme.json tokens (Section 3)the base tokens.css defaults
Page overridea token redefined on a page wrapper / scopethe tenant theme :root
Block overridean explicit prop on the block (e.g. bg=)everything — the token is only its fallback

In one line: tenant theme (:root) < page override < block override. Because tokens are additive fallbacks (Section 1), an un-migrated block that still passes an explicit colour just keeps working.

Proposal · Architecture

Shared workspace packages

Two apps, one copy of the code that matters. Move the data-access layer into packages/db and the reusable UI into packages/components, so glue-main and glue-composer consume the same source instead of maintaining drifting duplicates.

Proposed — awaiting approval. Nothing implemented yet.

Why

Today the MongoDB client (startMongo.ts) and the data-access files are duplicated across glue-main and glue-composer, and the two copies have already started to drift. Meanwhile the reusable UI components live only inside the published @clouda-inc/glue-components library, separate from the apps that use them.

Multi-tenancy makes this worse: the tenant-scoping guarantee (every query filtered by tenantId) has to be identical everywhere. Two copies means two places to get it right — and two places to get it wrong.

The duplication is the risk. If the scoping logic lives in two files, one of them will eventually miss a filter and leak a tenant's data. There must be exactly one home for the isolation guarantee.

The fix: move shared code into workspace packages so both apps import one copy.

Workspace layout

The monorepo already uses npm workspaces. Add a packages/* glob alongside the two apps so any package created under packages/ is automatically linked into both.

package.json (repo root)json
{
  "name": "clouda-monorepo",
  "private": true,
  "workspaces": [
    "packages/*",        // new — auto-links anything under packages/
    "glue-components",
    "glue-main",
    "glue-composer"
  ]
}
One copy of the shared code at the centre; both apps import it — nothing to keep in sync by hand.

packages/db

The shared data-access layer. It is the single home for the isolation guarantee and consists of three pieces:

mongo.ts

One merged startMongo — the connection/cache logic that today lives duplicated in both apps.

tenants.ts

getTenantBySlug / getTenantByDomain — the only un-scoped queries in the codebase.

repository.ts

getTenantRepo(tenant) force-injects tenantId on every read and write.

Both apps import from @clouda/db, so the duplication disappears and there is exactly one place the scoping rules live.

usage — glue-main & glue-composerts
// same import in BOTH apps — one copy of the logic
import { getTenantBySlug, getTenantRepo } from "@clouda/db";

const tenant = await getTenantBySlug(process.env.TENANT_SLUG!);
const repo = getTenantRepo(tenant);

// tenantId is force-injected — callers never pass it, never forget it
const pages = await repo.webpageContent.find({ pagePath, status: "live" });
CI gate. Pair the package with an ESLint / CI rule that forbids raw collection access (importing the Mongo client or calling db.collection(...)) anywhere outside packages/db. That makes the scoping layer impossible to bypass by accident.

packages/components

Move the reusable UI into packages/components for reuse and consistency across both apps. This is the evolution of @clouda-inc/glue-components into a workspace package — the same ~64 components, now living beside the apps that consume them rather than only as a published artifact.

Expose the public surface through exports subpaths, including a dedicated /theme entry for design tokens (the CSS-variable tokens described on the Theming page), so an app can import components and theme tokens independently.

packages/components/package.jsonjson
{
  "name": "@clouda/components",
  "version": "0.0.0",
  "exports": {
    ".": "./src/index.ts",
    "./theme": "./src/theme/index.ts"
  }
}
Both apps consume it. glue-main renders tenant sites from these components; glue-composer renders the same components in its editor preview — so the CMS shows exactly what visitors will see.

Wiring into the apps

Consume the workspace packages without a publish step during development: add them to transpilePackages and mirror the existing @clouda-inc/glue-components webpack alias already present in the Next config. Next.js then compiles the package source directly from the workspace.

glue-main/next.config.mjs & glue-composer/next.config.mjs (ESM, matching the real files)js
import path from "path";
import { fileURLToPath } from "url";
const __dirname = path.dirname(fileURLToPath(import.meta.url));

/** @type {import('next').NextConfig} */
const nextConfig = {
  // compile TS/JSX from the workspace packages — no build/publish needed
  transpilePackages: ["@clouda/db", "@clouda/components"],

  webpack(config) {
    // mirror the existing @clouda-inc/glue-components alias already in these files
    config.resolve.alias["@clouda/components"] =
      path.resolve(__dirname, "../packages/components/src");
    config.resolve.alias["@clouda/db"] =
      path.resolve(__dirname, "../packages/db/src");
    return config;
  },
};

export default nextConfig;
Result. Edit a file in packages/db or packages/components and both apps pick it up on the next reload — no version bump, no CodeArtifact publish in the inner dev loop.

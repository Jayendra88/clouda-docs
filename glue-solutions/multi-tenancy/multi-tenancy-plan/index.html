Multi-Tenancy for Clouda Glue Solutions

Run many tenant websites from one codebase — the same public app deployed per domain, one composer with a post-login tenant selector, per-tenant themes, and component visibility scoped global / shared / tenant-specific.

Context

Clouda wants to run many tenant websites from one codebase: the same glue-main app deployed to multiple domains (one per tenant), a single glue-composer where an editor picks a tenant after login and manages that tenant's pages, a component library that can mark components as tenant-specific / shared-with-some / global, and completely different themes per tenant.

Today the platform is strictly single-tenant:

  • Monorepo (npm workspaces): glue-components (published @clouda-inc/glue-components, ~64 components), glue-main (Next 14 public site, Amplify → clouda.io), glue-composer (Next CMS, Cognito + NextAuth).
  • One MongoDB glue DB shared by both apps; collections webpages + webpage_content; no tenantId anywhere. getGlobalSection()/getHomePage() return the first matching doc unfiltered — a direct cross-tenant hazard once data is shared.
  • block.name → component via glue-main/src/components/CloudaComponent.tsx.
  • Theming = per-page inline colors plus ~500 hardcoded brand hex (#F96F2D, etc.) across 80+ component files; global CSS class names; empty Tailwind color config.
  • Composer auth = Cognito groups → flat global roles (super-admin|admin|editor|viewer); the palette lists all 64 components flat from dist/cms/schema.json.
Decisions confirmed with the user: (1) Shared DB + tenantId discriminator; (2) deploy-per-tenant for glue-main, code kept host-resolution-capable; (3) foundation-first theming then incremental component waves; (4) MVP-first — get one real second tenant fully live before building the heavier admin/registry surface.

Outcome: a tenant-aware platform where a new tenant = one templated Amplify deploy + one tenants row + a theme, with complete data isolation enforced at the data layer.

Architecture at a glance

Isolation

Every tenant-owned doc carries tenantId (the tenant slug). All DB access flows through a mandatory tenant-scoping repository that force-injects tenantId on every read and write — an un-scoped query becomes structurally impossible. A dbName field is the escape hatch to DB-per-tenant later.

Shared data package

Extract packages/db to kill the duplicated startMongo.ts and near-duplicate data-access files across both apps, and host the repository + migrations in one enforced place.

Tenant resolution

resolveTenant() reads TENANT_SLUG (primary, deploy-per-tenant) and falls back to the Host header (kept for a future consolidated deploy / local preview).

Theming

Semantic design tokens (--glue-* CSS variables) injected server-side per tenant, with Tailwind mapped to those vars. Precedence: tenant theme < page override < block override.

Composer tenancy

One shared composer; tenant chosen post-login, stored in an httpOnly cookie validated against Mongo tenant_users memberships. A single getTenantContext() is the only source of tenantId — never the request body.

Component scoping = metadata

One shared library; a component_registry collection marks each component global / tenant-list / tenant-specific. Missing row ⇒ global (safe default). No per-tenant forks.

0Data foundation & shared repository

Goal: introduce tenancy plumbing while www.clouda.io behaves exactly as today. No behavior change.

  1. Create packages/db workspace; add "packages/*" to root workspaces; wire into both apps via transpilePackages + webpack alias (mirroring the existing glue-components setup).
    • mongo.ts — one merged startMongo (keep composer's serverApi options + dev HMR guard).
    • tenants.ts — getTenantBySlug / getTenantByDomain (the only un-scoped lookups; they are the scoping source).
    • repository.ts — getTenantRepo(tenant); the scoped wrapper spreads { tenantId } last on every filter and forces it on every insert/replace; never returns a raw collection.
  2. Seed the tenants collection + default tenant clouda (domains, primaryDomain, baseUrl, theme ref, analytics, integrations, status, dbName:"glue"). Unique indexes on slug and domains. Secrets are not stored plaintext here.
  3. Add tenant_users + component_registry collections now so migrations are one pass.
  4. Build tenantId-leading compound indexes before backfill: webpage_content {tenantId, pagePath, status}, {tenantId, parentId}; webpages {tenantId, type, status}. Partial-unique {tenantId, pagePath} where status:"live".
  5. Idempotent, reversible backfill script: set tenantId:"clouda" on existing docs where absent; verify count of un-tenanted docs is 0. Ship code first, then backfill; a transitional $or keeps un-migrated docs visible during the window.
  6. ESLint + CI gate: forbid importing startMongo or referencing raw collections outside packages/db — a forgotten filter becomes a build failure.
Files: packages/db/src/{mongo,tenants,repository}.ts, root package.json, glue-main/next.config.mjs, glue-composer/next.config.*

1glue-main: tenant resolution, scoped reads, per-tenant config

Goal: the public site is fully tenant-driven but identical for clouda.

  1. resolveTenant(host?) — return getTenantBySlug(process.env.TENANT_SLUG) when pinned; else getTenantByDomain(host); else 404. Wrapped in React cache(). Set TENANT_SLUG=clouda on the existing Amplify app.
  2. Extend middleware.ts to also set x-tenant-host from the Host header (unused while env-pinned; ready for host mode).
  3. Refactor data access to take a tenant and go through getTenantRepo; thread it through layout, pages, sitemap, robots, draft. Fixes the unfiltered getGlobalSection()/getHomePage() and the full-collection scan.
  4. Move hardcoded config to the tenant record: GTM, Zoho SalesIQ, PageSense, marketing pixel, metadataBase (also fixes the existing .com vs .io mismatch), sitemap/robots base URL, and the draft secret. Real secrets go to AWS SSM / Secrets Manager, not Mongo.
  5. generateStaticParams(tenant) scopes to the pinned tenant — each build/CDN contains only that tenant's HTML (secondary isolation boundary).
Files: glue-main/src/app/layout.tsx, app/[...slug]/page.tsx, src/lib/{webpageContent,webpages,constants,tenant}.ts, src/middleware.ts, app/{sitemap,robots}.ts, app/api/draft/route.ts

2Theming foundation

Goal: a new tenant gets a visibly distinct theme immediately; component refactor comes in waves after.

  1. Token layer in the library: theme/tokens.css (default :root{--glue-*} = current brand, so un-themed render is unchanged) + tokens.ts (ThemeTokens type). Colors as RGB channels so opacity utilities like bg-primary/40 work.
  2. Map Tailwind to the vars in all three configs — additive, keeping existing content / screens / transitions. Existing utility classes become theme-reactive without editing JSX.
  3. Server-side injection in layout.tsx: emit <style id="glue-theme">:root{--glue-...}</style> from the resolved tenant's theme (no FOUC), load tenant fonts, set favicon.
  4. Preserve precedence: tenant vars < page override < block override. Tokens are a new fallback layer underneath the existing props — no signature change to CloudaComponent.tsx or the layout initially.
  5. themes collection (referenced by tenants.themeId): tokens, fonts, logo. Editing UI arrives in Phase 6.
Files: glue-components/theme/{tokens.css,tokens.ts}, index.css, all three tailwind.config.*, glue-main/src/app/layout.tsx, glue-main/src/lib/theme.ts

3Composer: tenant context, per-tenant auth, scoped operations

Goal: one composer where a user selects a tenant and can only touch that tenant's content.

  1. Tenant context: getTenantContext() → { tenantId, tenantSlug, effectiveRoles, isGlobalSuperAdmin }, from an httpOnly active_tenant cookie validated against JWT memberships. Extend the NextAuth callbacks to load tenant_users memberships. Add POST /api/tenant/select + /api/tenant/list and a client TenantProvider.
  2. Access model = Mongo tenant_users {userId, tenantId, roles[], status} as source of truth; Cognito keeps only a global super-admin group. requirePermission becomes tenant-aware and returns the resolved tenantId; add requireTenantAccess. serverCan/usePermission read effective per-tenant roles (signatures unchanged).
  3. Scope every content op: add tenantId to every filter and insert; getWebpageById returns null for a foreign tenant (kills IDOR); the create path stamps tenantId from context.
  4. Close pre-existing security gaps: PUT /api/webpageContent/[id] and /api/webpage/list are currently unguarded; db-snapshot/create dumps every collection unscoped — restrict to global super-admin.
  5. Middleware: no valid active tenant on a scoped route → redirect to /select-tenant; gate /admin to global super-admin.

Composer UX (Phase 3 deliverables)

  • /select-tenant chooser post-login (auto-select for single-tenant users).
  • Persistent tenant switcher chip in the header, tinted with the tenant's brand color.
  • Mid-edit guardrail: switching tenant with unsaved changes prompts confirm/discard.
  • Per-tenant page list + empty states; publish/delete confirmations restate the tenant name.
Files: glue-composer/src/lib/{tenantContext,webpages,webpageContent,apiGuard,serverPermission}.ts, hooks/usePermission.ts, app/utils/getSession.ts, types/next-auth.d.ts, middleware.ts, app/api/{tenant,webpage,webpageContent}/**

4MVP milestone: onboard a real second tenant

Goal: prove the whole loop end-to-end.

  1. Reusable IaC (CDK or Terraform Amplify module) parameterized by tenantSlug, domains, secret ARNs; reuse the existing amplify.yml unchanged.
  2. Onboard tenant #2: tenants row + themes row (distinct palette/fonts) + SSM secrets + one Amplify app instance with its domain and TENANT_SLUG.
  3. Composer loop: super-admin creates the tenant, assigns an editor, the editor selects it, builds pages from global components, and the site renders on the new domain with its own theme.
  4. ISR + on-demand revalidation (closes a gap where publishing doesn't update the live site until a rebuild): a secured POST /api/revalidate invoked by composer publish.

Later phases — documented now, built after MVP

Phase 5 — Component library at scale

Replace the monolithic cms/schema.json (~11.8k lines) with per-component manifests co-located with each component (schema + meta{category, tags, thumbnail, status, version, defaultVisibility}). A Rollup prebuild aggregates them into a byte-compatible dist/cms/schema.json (composer contract unchanged) plus a new dist/cms/manifest.json. Add a manifest lint; keep the single package, add exports subpaths and move toward named exports for tree-shaking.

Phase 6 — Component visibility + palette + admin UIs

component_registry visibility resolution (getVisibleComponents(tenantId)); rebuild the flat 64-item list into a searchable, category-grouped palette (validate blocks on save). src/app/admin/** for global super-admin: tenants CRUD, members & roles, component-visibility matrix, and the theme editor with live iframe preview.

Phase 7 — Component theming waves

Codemod + human review to migrate ~500 hardcoded hex / ~1000 color-prop usages to tokens, prioritized chrome → high-use cards → long tail; migrate global CSS to CSS Modules to remove collision risk; retire the isLightTheme hex-guessing and AutoSwapColor legacy paths last.

Phase 8 — Backup / restore hardening

Per-tenant page backup/restore filtered by tenantId; restore forces target tenantId and rejects mismatches; full-DB snapshot stays global-super-admin only.

Risks & mitigations

1
Cross-tenant data leakageCriticalMandatory getTenantRepo forces tenantId (spread last) on every read/write; no raw .collection() outside packages/db; ESLint + CI gate; tenantId-leading indexes; deploy-per-tenant artifact isolation. Composer derives tenantId only from getTenantContext().
2
Partial / failed backfill hides pagesHighAdditive, idempotent, $exists:false-guarded; indexes before backfill; transitional $or default match; verify count == 0; reversible (drop field/indexes reverts to single-tenant).
3
SSG blow-up / stale cacheHighDeploy-per-tenant scopes generateStaticParams; ISR + on-demand revalidatePath on publish; per-domain CDN naturally isolated.
4
Duplicated data-access driftMediumEliminated by packages/db; apps keep only thin serialization; lint forbids direct DB access.
5
Pre-existing security holesHighUnguarded PUT /api/webpageContent/[id] & /api/webpage/list; unscoped snapshot dump; hardcoded draft secret; .com/.io metadata mismatch — all fixed in Phases 1/3.
6
Theming regression across ~500 hex sitesMediumFoundation-first + tokens-as-fallback (never remove a working prop path in the same change); prioritized waves; publish between waves; CSS Modules migration in parallel.
7
Stale tenant contextMediumJWT maxAge 3600 vs membership change — re-validate membership at the data/guard layer on sensitive writes; invalid active tenant bounces to /select-tenant.
8
Host spoofing (consolidated mode only)MediumEnv-pin ignores Host; host mode validates against unique tenants.domains and 404s on miss.
9
Per-tenant deploy ops costMediumIaC module keeps a new tenant to one templated instance; revisit host-based consolidation beyond dozens of tenants (code already supports it).

UX suggestions

  • Post-login tenant chooser with tenant cards (logo, brand tint, your role); auto-select for single-tenant users.
  • Always-visible active-tenant chip in the composer header, tinted with the brand color; tenant name in the tab title — so no one edits the wrong site.
  • Mid-edit switch guardrail with unsaved-changes confirmation.
  • Searchable, category-grouped component palette (Phase 6) replacing the flat 64-item list, with tenant-specific badges, thumbnails, keyboard nav, and a recently-used row.
  • Theme editor with live preview (Phase 6): token form on the left, iframe rendering a representative page with draft tokens on the right, device toggle, draft-vs-published, and a "view on live domain" link.
  • Confirmations restate the tenant name on publish/delete; clear empty states.

Verification

  • Isolation tests: create pages under tenant A, assert tenant B's scoped ops return nothing / no-op; cross-tenant _id fetch → 404; assert every insert stamps tenantId.
  • Lint gate: CI fails if any file outside packages/db imports startMongo or references raw collections.
  • Backfill dry-run against a snapshot; assert un-tenanted count == 0 and clouda page counts unchanged.
  • glue-main parity: with TENANT_SLUG=clouda, diff rendered HTML of key pages before/after Phases 1–2 (should be byte-identical); confirm GTM/Zoho/canonical/sitemap now derive from the tenant record.
  • Second-tenant E2E (MVP): onboard tenant #2, build + publish a page, confirm it renders on the new domain with its own theme and that tenant #1's content/theme never appear; confirm revalidation updates without a rebuild.
  • Auth: requirePermission returns the resolved tenantId; a user without a membership for tenant X cannot read/write X; global super-admin can access all tenants; the two previously-unguarded routes now reject unauthorized/cross-tenant calls.
  • Theming: render a component under two themes via --glue-* overrides; confirm page/block color overrides still win over tenant defaults.
Sequencing: Phases 0–4 are the MVP and each step is independently shippable and single-tenant-safe — www.clouda.io behaves identically until tenant #2 is onboarded in Phase 4. Phases 5–8 harden the platform in parallel tracks afterward.

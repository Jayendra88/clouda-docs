Per-tenant theming with a shared component library

How one shared library of components can render a completely different look, feel and behaviour per tenant — including headers whose colours, shapes, hover effects and scroll-on-top behaviour differ — without forking components per tenant. Companion to index.html / plan.md (Phases 2 & 7).

The layered model

The mistake most teams make is trying to solve theming with one mechanism (usually a pile of colour props). Different tenants differ in colour, in shape, and in behaviour — and those need different tools. We use five layers, applied top-to-bottom, each handling more of the divergence than the one below it, so per-tenant work stays tiny and the shared code stays generic.

Reach for the highest layer that solves the problem. ~80% of tenant divergence is Layer 1 alone.
LayerHandlesMechanismPer-tenant authoring
1 · TokensColour palette, fonts, corner radius, spacing, shadowsCSS custom properties (--glue-*) + Tailwind mapped to themData (theme record) — no code
2 · VariantsStructural shape/layout choices (flat vs boxed vs pill header)A typed variant the component reads and maps to classesData (pick from an enum)
3 · BehaviourHover style, scroll-on-top, motion, sticky logicBehaviour flags in a ThemeProvider context; component branches on themData (pick from an enum)
4 · OverridesOne-off bespoke tweaks for a single tenantScoped CSS [data-tenant="x"]{…} overriding tokens/classesA small CSS string (theme editor / file)
5 · ComponentsA block only one tenant needs, or a radically unique oneA normal library component, gated by the component registryCode (rare) + a registry row

1How the theme changes based on the tenant

Because we deploy one build per tenant, resolution is trivial and there is no per-request lookup: the running site already knows which tenant it is from an env var. That tenant's theme is fetched once, then applied two ways — tokens injected as CSS variables server-side (colours/shape, no flash) and variant/behaviour flags provided via React context (structure/behaviour).

The flow

  1. resolveTenant() reads process.env.TENANT_SLUG → loads the tenant + its theme record.
  2. The root layout.tsx serialises the theme's tokens into a <style> block in <head> and stamps data-tenant / data-theme on <html>.
  3. A ThemeProvider puts the non-CSS parts (variants, behaviour flags, asset URLs) on React context.
  4. Components read tokens through CSS/Tailwind, and read variants/behaviour through useTheme().
glue-main/src/app/layout.tsx (server component)tsx
import { resolveTenant } from "@/lib/tenant";
import { themeToCssVars } from "@/lib/theme";
import { ThemeProvider } from "@clouda-inc/glue-components/theme";

export default async function RootLayout({ children }) {
  const tenant = await resolveTenant();      // from TENANT_SLUG (or Host)
  const theme  = tenant.theme;                // tokens + variants + behaviour + assets

  const cssVars = themeToCssVars(theme.tokens); // ":root{--glue-color-bg:#0a0a0a; ...}"

  return (
    <html lang="en" data-tenant={tenant.slug} data-theme={theme.name}>
      <head>
        {/* Server-rendered = no flash of the wrong theme (FOUC) */}
        <style id="glue-theme" dangerouslySetInnerHTML={{ __html: cssVars }} />
        {theme.fonts.link && <link rel="stylesheet" href={theme.fonts.link} />}
      </head>
      <body>
        {/* variants + behaviour + assets travel via context, not CSS */}
        <ThemeProvider value={theme}>{children}</ThemeProvider>
      </body>
    </html>
  );
}
glue-main/src/lib/theme.ts — tokens object → CSS stringts
export function themeToCssVars(tokens: Record<string, string>): string {
  const body = Object.entries(tokens)
    .map(([name, value]) => `--glue-${name}:${value};`)
    .join("");
  return `:root{${body}}`;
}
Why server-side injection? If tokens loaded on the client, the first paint would use the default theme, then visibly repaint into the tenant's theme (FOUC). Emitting the variables in the server-rendered <head> means the correct colours are there on first paint.

2The token contract

Tokens are the single source of truth for the visual language. Components reference semantic tokens (surface, primary) — never raw hex, never a specific brand colour. The default values equal today's Clouda brand, so nothing changes until a tenant overrides them.

glue-components/theme/tokens.css — defaults on :rootcss
:root{
  /* colours — RGB *channels* so Tailwind's /alpha works: bg-primary/40 */
  --glue-color-bg:            10 10 10;
  --glue-color-fg:            255 255 255;
  --glue-color-primary:       249 111 45;   /* Clouda orange */
  --glue-color-primary-fg:    255 255 255;
  --glue-color-accent:        249 111 45;
  --glue-color-surface:       17 19 23;
  --glue-color-surface-fg:    232 234 237;   /* paired fg — no "guess light/dark" */
  --glue-color-border:        38 41 47;
  /* shape */
  --glue-radius-sm: 4px;  --glue-radius-md: 10px;  --glue-radius-pill: 999px;
  /* type */
  --glue-font-sans:    Inter, system-ui, sans-serif;
  --glue-font-heading: var(--glue-font-sans);
  /* header-specific shape tokens (used by the worked example) */
  --glue-header-height: 78px;
  --glue-header-radius: 0px;
}

Map Tailwind to the tokens so the thousands of existing utility classes become theme-aware with no JSX rewrite. text-fg, bg-surface, rounded-md now follow the tenant.

tailwind.config.js (shared across all three configs)js
theme: {
  extend: {
    colors: {
      bg:         "rgb(var(--glue-color-bg) / <alpha-value>)",
      fg:         "rgb(var(--glue-color-fg) / <alpha-value>)",
      primary:    "rgb(var(--glue-color-primary) / <alpha-value>)",
      "primary-fg": "rgb(var(--glue-color-primary-fg) / <alpha-value>)",
      accent:     "rgb(var(--glue-color-accent) / <alpha-value>)",
      surface:    "rgb(var(--glue-color-surface) / <alpha-value>)",
      "surface-fg": "rgb(var(--glue-color-surface-fg) / <alpha-value>)",
    },
    fontFamily: { sans: ["var(--glue-font-sans)"], heading: ["var(--glue-font-heading)"] },
    // distinct keys — do NOT reuse sm/md or you silently remap Tailwind's built-in rounded-sm/md everywhere
    borderRadius:{ "glue-sm":"var(--glue-radius-sm)", "glue-md":"var(--glue-radius-md)", pill:"var(--glue-radius-pill)" },
  }
}
Rule of thumb: in a component, bg-surface text-surface-fg rounded-md instead of bg-[#111317] text-white rounded-[10px]. Same markup renders differently per tenant, for free.

3Components for different themes — colours & shapes

The bulk of "this header looks different" is colour and shape, and Layer 1 handles all of it. The job is to replace hardcoded values with token references. Here is your real HeaderMain pattern, before and after.

Before — hardcoded (glue-components/components/header-main)tsx
// colour decided by a brittle MULTI-clause hex-equality "is it light?" guess
const isLightTheame =
  customBackgroundColor === "#ffffff" || customBackgroundColor === "#fff" ||
  customForegroundColor === "#000" || customForegroundColor === "black"; // ...5 clauses in the real file
// hardcoded brand orange, hardcoded black gradient
borderBottom: "2px solid #F96F2D",
background: "linear-gradient(180deg, #000000 16.74%, rgba(0,0,0,0) 99.42%)",
After — token-driven (works for every tenant)tsx
// no guessing: the theme supplies a paired foreground for the header surface.
// precedence preserved — an explicit prop still WINS; the token is the fallback.
<div
  className="headerMain"
  style={{
    background: headerBgColorOverride ?? "var(--glue-header-bg, rgb(var(--glue-color-bg)))",
    color:      headerFgColorOverride ?? "var(--glue-header-fg, rgb(var(--glue-color-fg)))",
    borderRadius: "var(--glue-header-radius)",
  }}
>
  {/* the open-menu underline uses the accent token, not #F96F2D */}
  <div style={{ borderBottom: "2px solid rgb(var(--glue-color-accent))" }} />
</div>

Now two tenants produce two headers from the same component just by shipping different token values:

Tenant A — dark, sharp, orange

  • --glue-color-bg: 10 10 10
  • --glue-color-accent: 249 111 45
  • --glue-header-radius: 0px
  • --glue-font-sans: Inter

Tenant B — light, rounded, teal

  • --glue-color-bg: 255 255 255
  • --glue-color-accent: 14 165 165
  • --glue-header-radius: 16px
  • --glue-font-sans: "Poppins"
Retire the light/dark guess. The current isLightTheame hex-equality check and AutoSwapColor.ts (which inverts a hex) are fragile — they break for any colour that isn't pure black/white. Replace them with explicit paired tokens (surface + surface-fg): the theme states its own foreground, so the component never has to infer contrast.

4Shape / layout differences — component variants

Some differences aren't a single value — they're a structural choice: a flat underline nav vs. a boxed nav vs. a rounded pill nav. Model these as a small, typed variant the theme selects, which the component maps to a set of classes. This keeps one component instead of three near-duplicates.

glue-components/theme/types.ts — the variant contractts
export type HeaderShape = "flat" | "boxed" | "pill";

export interface ThemeVariants {
  header: { shape: HeaderShape };
  card:   { shape: "square" | "rounded"; elevation: "flat" | "raised" };
  // ...one entry per themable component
}
Component maps variant → classes (data-attribute keeps CSS declarative)tsx
import { useTheme } from "../theme";

const { variants } = useTheme();
return <header className="headerMain" data-shape={variants.header.shape}> ... </header>;
glue-components/components/header-main/headerMain.csscss
.headerMain[data-shape="flat"]  .navItem { border-radius: 0; }
.headerMain[data-shape="boxed"] .navItem { border-radius: var(--glue-radius-md);
                                       background: rgb(var(--glue-color-surface)); padding: 8px 14px; }
.headerMain[data-shape="pill"]  .navItem { border-radius: var(--glue-radius-pill);
                                       padding: 8px 18px; }
Why an enum, not free-form CSS? A closed set of variants is testable (Storybook renders all three), typed (a theme can't request a shape that doesn't exist), and keeps the design coherent. Free-form per-tenant CSS is Layer 4 — the escape hatch, used sparingly.

5Behaviour differences — hover & scroll-on-top

This is the part pure CSS can't fully cover. Your header has real behaviour: an animated underline on hover, and a "transparent gradient at the top that solidifies once you scroll" effect driven by react-headroom + a scroll listener. Different tenants may want different behaviour (underline vs. fill hover; fade-in vs. always-solid vs. hide-on-scroll). Model behaviour as flags the component branches on — CSS where possible, JS where necessary.

glue-components/theme/types.ts — behaviour contractts
export type HeaderHover  = "underline" | "fill" | "none";
// fade   = transparent at top, solidifies on scroll (today's Clouda behaviour)
// solid  = always opaque
// hide   = slides away scrolling down, returns scrolling up
// shrink = stays, reduces height on scroll
export type HeaderScroll = "fade" | "solid" | "hide" | "shrink";

export interface ThemeBehaviour {
  header: { hover: HeaderHover; scroll: HeaderScroll; motion: "full" | "reduced" };
}

A · Hover — mostly CSS, switched by a data-attribute

headerMain.css — hover behaviour per flagcss
/* underline: the width 0 → 100% animation you already have */
.headerMain[data-hover="underline"] .navItem::after{
  content:""; position:absolute; left:0; bottom:0; height:2px; width:0;
  background: rgb(var(--glue-color-accent)); transition: width .35s ease;
}
.headerMain[data-hover="underline"] .navItem:hover::after{ width:100%; }

/* fill: pill fills with accent on hover */
.headerMain[data-hover="fill"] .navItem{ transition: background .2s ease, color .2s ease; }
.headerMain[data-hover="fill"] .navItem:hover{
  background: rgb(var(--glue-color-accent)); color: rgb(var(--glue-color-primary-fg));
}

/* honour the reduced-motion behaviour flag AND the OS setting */
@media (prefers-reduced-motion: reduce){
  .headerMain .navItem::after{ transition:none; }
}

B · Scroll-on-top — JS behaviour, gated by the flag

The scroll effect needs JavaScript, so the component keeps all behaviours and picks one from the flag. Today's "transparent-at-top, solid once you scroll" effect is produced by two cross-fading overlay layers (.gradientOverlay) — independent of react-headroom, which the current header engages unconditionally to hide/pin on scroll. We surface that overlay effect as the "fade" branch and add "solid", "hide" and "shrink" as alternatives.

HeaderMain.tsx — behaviour selected from theme, not hardcodedtsx
const { behaviour } = useTheme();
const { hover, scroll } = behaviour.header;

// scroll/hover state that drives the "fade" branch (rAF listener, as in today's code)
const [isAtTop, setIsAtTop]   = useState(true);
const [isHovered, setHovered] = useState(false);
useEffect(() => {
  if (scroll !== "fade") return;                 // only "fade" needs a scroll listener
  const onScroll = () => requestAnimationFrame(() => setIsAtTop(window.scrollY < 78));
  window.addEventListener("scroll", onScroll, { passive: true });
  return () => window.removeEventListener("scroll", onScroll);
}, [scroll]);

// NOTE: today HeaderMain engages react-headroom UNCONDITIONALLY (pin/hide on scroll).
// Making it a per-theme choice — only "hide" pins, others disable the library — CHANGES
// today's default, so this part is not purely additive. Call it out when migrating.
const headroomProps = scroll === "hide"
  ? { pinStart: 78, upTolerance: 5, downTolerance: 10 }
  : { disable: true };

// "fade" cross-fade opacity — matches the real at-rest condition (incl. menu state)
const atRest = isAtTop && !isHovered && !isMenuOpen && !isSwitching;
const solidOpacity = scroll === "fade" ? (atRest ? 0 : 1) : 1;

return (
  <Headroom {...headroomProps}>
    <header
      className="headerMain"
      data-hover={hover}
      data-scroll={scroll}
      data-shrink={scroll === "shrink" && !isAtTop ? "true" : undefined}
    >
      {/* real HeaderMain stacks TWO .gradientOverlay divs (solid + gradient) that cross-fade */}
      {scroll === "fade" && (
        <div className="gradientOverlay" style={{ opacity: solidOpacity }} />
      )}
      {/* ...nav... */}
    </header>
  </Headroom>
);
Not purely additive: today's header always engages react-headroom (hide/pin on scroll) and does the transparent→solid effect with overlay opacity. Turning scroll behaviour into a per-theme choice changes that default for tenants that pick "fade"/"solid" — flag it as a deliberate behaviour change during migration, not a silent addition.
headerMain.css — shrink handled declarativelycss
.headerMain{ height: var(--glue-header-height); transition: height .3s ease; }
.headerMain[data-shrink="true"]{ height: calc(var(--glue-header-height) - 22px); }
.headerMain[data-scroll="solid"]{ background: rgb(var(--glue-color-bg)); } /* no gradient */
Principle: behaviour is a closed set of flags, not arbitrary per-tenant logic. Every branch lives in the shared component and is covered by a story/test. A tenant chooses a behaviour; it never ships behaviour code.

The Layer-4 override is injected after the tokens and scoped so it can only affect this tenant's site (and, in the composer's preview, only the previewed tenant):

layout.tsx — scoped override, tokens first then overridetsx
<style id="glue-theme" dangerouslySetInnerHTML={{ __html: themeToCssVars(theme.tokens) }} />
{theme.cssOverride && (
  <style id="glue-tenant-override"
    dangerouslySetInnerHTML={{ __html: `[data-tenant="${tenant.slug}"]{ ${theme.cssOverride} }` }} />
)}
Deploy-per-tenant bonus: because each tenant is its own build, only that tenant's override ships to that site. In the composer (one app, many tenants) the [data-tenant] scope keeps a preview of Tenant A from leaking Tenant B's overrides.

8Keeping a completely different look & feel per tenant

"Completely different" is the sum of the layers plus a few whole-site levers. A tenant's identity = tokens + variants + behaviour + fonts/logo + (optional) overrides. In practice that is enough for sites that look nothing alike while sharing 100% of the component code.

  • Palette & type (tokens): different colours; fonts loaded per tenant — via next/font when the family is fixed at build time (fine under deploy-per-tenant), or a runtime <link>/@import from the theme record when it must be data-driven.
  • Shape language (tokens + variants): radius scale, nav/card shapes, density.
  • Motion & interaction (behaviour): hover, scroll, reduced-motion.
  • Brand assets (theme record): logo (light/dark), favicon, OG image.
  • Structure (optional): a tenant can prefer different templates (you already have Two-Column, Left-Sticky, Right-Sticky) and a different default component set via the registry.
  • Bespoke polish (Layer 4): the scoped CSS override for the last 5%.
Where to draw the line. If two tenants need structurally different headers that no variant can express, that's a signal to add a new variant to the shared component — not to fork HeaderMain into HeaderMainAcme. Forks multiply maintenance across every future change and are the fastest way to the "mess" you want to avoid.

9Managing a large library — all components, all tenants, one app

One shared library serves every tenant. Scale is managed by organisation + contracts + tests, and by preferring variants over forks so the component count grows with genuine capability, not with tenant count.

9.1 Co-located, self-describing components

Replace the single 11.8k-line cms/schema.json with a manifest beside each component; a build step aggregates them. Each manifest carries the metadata that keeps a big catalogue navigable.

components/header-main/HeaderMain.manifest.tsts
export default {
  name: "HeaderMain",                 // must equal the index.ts export key
  category: "Navigation",             // powers grouped, searchable palette
  tags: ["header", "menu", "nav"],
  status: "active",                    // active | beta | deprecated
  variants: { shape: ["flat", "boxed", "pill"] },   // declared for Storybook + validation
  defaultVisibility: "global",        // global | tenant-list | tenant-specific
  schema: { /* existing rjsf JSON Schema of props */ },
} satisfies ComponentManifest;

9.2 Tenant scoping is metadata, not code

A "tenant-specific component" is a normal component whose component_registry row limits which tenants see it in the palette. The renderer is unaffected; visibility gates insertion, not runtime. This is how you avoid per-tenant code branches. (Detail in index.html Phases 5–6.)

9.3 Guardrails that keep it tidy as it grows

  • No hardcoded colours — a lint rule fails the build on raw hex in component source, forcing token use.
  • Naming lint — folder ↔ PascalCase ↔ manifest name must agree; names unique; category from an allowed set.
  • Variants > forks — add a variant to an existing component before creating a new one; PR review enforces this.
  • Deprecation, not deletion — status:"deprecated" hides a component from new inserts but keeps existing pages rendering.
  • Tree-shaking — named exports + sideEffects so a tenant's build only bundles the components it uses.
.eslintrc — ban hardcoded colours in component sourcejs
rules: {
  "no-restricted-syntax": ["error", {
    selector: "Literal[value=/#(?:[0-9a-fA-F]{3}){1,2}\\b/]",
    message:  "Use a --glue-* token (or a Tailwind token class), not a hardcoded hex."
  }]
}

9.4 Testing across the theme matrix

  • Storybook theme switcher — a global decorator injects a chosen theme's tokens so every story is viewable under each tenant theme.
  • Visual regression (Chromatic / Playwright screenshots) over component × theme × variant catches a token change that breaks one tenant.
  • Contrast checks — assert each theme's paired fg/bg tokens meet WCAG contrast, so a new palette can't ship unreadable text.
.storybook/preview.tsx — theme decoratortsx
export const globalTypes = { theme: { toolbar: { items: ["clouda", "acme", "lumen"] } } };

export const decorators = [(Story, ctx) => {
  applyThemeVars(THEMES[ctx.globals.theme]);   // sets --glue-* on the preview root
  return <ThemeProvider value={THEMES[ctx.globals.theme]}><Story /></ThemeProvider>;
}];

10Other major concerns to address

ConcernHow it's handled
FOUC (flash of wrong theme)Tokens rendered server-side in <head>; no client theme swap on load.
PrecedenceFixed order: tenant theme (:root) < page override (wrapper vars) < block override (props). Tokens sit below existing colour props, so old pages keep working.
Light/dark within a themePaired *-fg tokens; never infer contrast from a hex (retire isLightTheame / AutoSwapColor).
AccessibilityContrast asserted per theme in CI; focus states use tokens; honour prefers-reduced-motion alongside the behaviour flag.
CSS class collisionsGlobal class names are a growing risk; migrate per-component CSS to CSS Modules (or a build-time prefix). Leave third-party selectors (.headroom*, slick) alone.
Bundle sizeOne library, tree-shaken via named exports; deploy-per-tenant ships only that tenant's overrides; fonts loaded per tenant.
Versioning & blast radiusOne package version affects all tenants — so change via additive variants/flags, not breaking edits; validate with the theme matrix; roll out to a canary tenant first.
PerformanceCSS variables are essentially free; behaviour flags avoid layout thrash (transform/opacity, not width/height where possible); scroll work stays in requestAnimationFrame as today.
GovernanceTheme editor with live preview (Phase 6); token completeness validated on save so a theme can't ship missing a required token.
Migration riskTokens are an additive fallback (prop ?? var(--token)); components migrate in waves; publish between waves; nothing big-bang.

Step-by-step rollout

  1. Land the token contract — tokens.css + types.ts + Tailwind mapping. Defaults = current brand ⇒ zero visual change. (Plan Phase 2)
  2. Server-inject tokens in layout.tsx from a default theme; verify the render is byte-identical.
  3. Add the ThemeProvider + useTheme() and the variant/behaviour contracts (empty/default to start).
  4. Migrate shared chrome first — HeaderMain, Footer, Title: tokens for colour/shape, then the hover/scroll behaviour flags. Retire isLightTheame/AutoSwapColor here.
  5. Migrate high-use components in waves (cards, CTAs), then the long tail. (Plan Phase 7)
  6. Add CSS Modules + the no-hardcoded-colour lint alongside the waves.
  7. Stand up the theme matrix — Storybook switcher + visual regression + contrast checks.
  8. Build the theme editor with live preview and the Layer-4 override field. (Plan Phase 6)
  9. Onboard the second tenant's theme — a record of tokens/variants/behaviour/assets; confirm two visibly different sites from one library.
Net effect: a new tenant's entire look is a data record plus, occasionally, a small scoped CSS override — no component forks, no per-tenant code branches, and a library that grows with real capability rather than with the number of tenants.

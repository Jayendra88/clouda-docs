Architecture · Content Management

Content management for many tenants

One Glue Composer serves every tenant. After login, users pick a tenant; from then on they can only create, update, and delete content for the tenant they selected. A single Super User manages tenants themselves — creating them, assigning members, and viewing tenant-level stats.

Proposed — awaiting approval. Nothing implemented yet.
The core rule. The active tenant is never trusted from the request body. It comes from a validated httpOnly cookie via one server helper, getTenantContext(), which is the single source of the tenant id for every read and write.

1One composer, tenant selection after login

Authentication stays exactly as it is today — NextAuth v4 backed by AWS Cognito. The change happens after a session exists. We look up which tenants the user belongs to and route accordingly:

  • Access to more than one tenant → redirect to a /select-tenant chooser.
  • Access to exactly one tenant → auto-select it and continue straight into the workspace.
  • Access to zero tenants → a friendly empty state (a Super User must assign membership first).

The chosen tenant is written to an httpOnly cookie named active_tenant. On every request that cookie is validated against the user's tenant memberships — a raw cookie value is never trusted on its own. Resolution is centralised in a new helper so no route ever re-implements it.

One entry, one validated cookie, one scoped workspace — regardless of how many tenants a user can reach.

The helper below lives at glue-composer/src/lib/tenantContext.ts. It returns everything a handler needs and is the only place the tenant id is derived:

glue-composer/src/lib/tenantContext.tsts
import { cookies } from "next/headers";
import { getSession } from "@/app/utils/getSession";
import { getUserMemberships } from "./tenants";

export type TenantContext = {
  tenantId: string;
  tenantSlug: string;
  effectiveRoles: Role[];        // the user's roles WITHIN this tenant
  isGlobalSuperAdmin: boolean;
};

export async function getTenantContext(): Promise<TenantContext> {
  const session = await getSession();
  if (!session) throw new Error("UNAUTHENTICATED");

  const raw = cookies().get("active_tenant")?.value;
  const memberships = await getUserMemberships(session.userId);

  // Never trust the cookie: it must match a real membership.
  const m = memberships.find((x) => x.tenantId === raw);
  if (!m) throw new Error("NO_ACTIVE_TENANT"); // -> /select-tenant

  return {
    tenantId: m.tenantId,
    tenantSlug: m.tenantSlug,
    effectiveRoles: m.roles,
    isGlobalSuperAdmin: session.isGlobalSuperAdmin === true,
  };
}

2Scoped create / update / delete

Every function in glue-composer/src/lib/webpages.ts and webpageContent.ts gains a required tenantId parameter that is folded into every Mongo filter and every insert. This is the wall that stops IDOR: getWebpageById(tenantId, id) returns null for a document that belongs to another tenant, so a guessed _id reveals nothing.

  • Reads: getAllWebpages(tenantId, …), getWebpageById(tenantId, id) — the tenant is part of the query.
  • Create: POST /api/webpage → createEmptyPage(tenantId) stamps the tenant on both the webpages and webpage_content documents.
  • Update / delete: the same tenantId filter is required, so a foreign document simply never matches.
Route handlers never read tenantId from the client body. They derive it from getTenantContext() and pass it down. The body describes what to change; the context decides whose data it is.
glue-composer/src/lib/webpages.tsts
export async function getAllWebpages(
  tenantId: string,
  filter: Partial<WebpageQuery> = {}
) {
  const query = { ...filter, tenantId };   // tenant folded into EVERY filter
  return db.collection("webpages").find(query).toArray();
}

export async function getWebpageById(tenantId: string, id: string) {
  // Foreign tenant -> no match -> null (blocks IDOR)
  return db.collection("webpages").findOne({ _id: toId(id), tenantId });
}

export async function createEmptyPage(tenantId: string, name: string) {
  const page = await db.collection("webpages").insertOne({
    name, type: "page", status: "draft", tenantId,   // stamped on insert
    createdAt: new Date(), updatedAt: new Date(),
  });
  await db.collection("webpage_content").insertOne({
    parentId: page.insertedId, name, status: "draft",
    tenantId, blocks: [], settings: [],          // tenant on content doc too
  });
  return page.insertedId;
}

3Roles and the Super User

The permission vocabulary in glue-composer/src/lib/permissions.ts is unchanged, and permissions stay a pure function of roles → capability. Tenancy is applied purely by which roles are passed in: a user's roles now come per-tenant from a Mongo tenant_users mapping. The per-tenant roles are admin, editor, and viewer; super-admin is not a per-tenant role — the cross-tenant Super User is the global isGlobalSuperAdmin flag (matching the Data Model page's tenant_users.roles).

CollectionShapePurpose
tenant_users{ userId, tenantId, roles[] }Which roles a user holds inside a given tenant.
Global flagisGlobalSuperAdmin on the sessionCross-tenant Super User; independent of any single tenant's roles.

The existing guards — apiGuard.ts requirePermission(), serverPermission.ts serverCan(), and hooks/usePermission.ts — become tenant-aware. Critically, requirePermission(perm) now returns the resolved tenantId so a handler can pass it straight into the scoped data functions above.

glue-composer/src/lib/apiGuard.ts · app/api/admin/tenants/route.tsts
// Tenant-aware guard: resolves context, checks the per-tenant roles,
// and RETURNS the tenantId for the handler to use.
export async function requirePermission(perm: Permission) {
  const ctx = await getTenantContext();
  if (!hasPermission(ctx.effectiveRoles, perm)) throw new Error("FORBIDDEN");
  return { tenantId: ctx.tenantId };
}

// Super-User-only admin route: create a tenant.
export async function POST(req: Request) {
  const ctx = await getTenantContext();
  if (!ctx.isGlobalSuperAdmin) return forbidden();

  const { slug, name } = await req.json();
  const tenant = await createTenant({ slug, name });
  return Response.json(tenant, { status: 201 });
}

The Super User (isGlobalSuperAdmin) sits above the per-tenant model. They can create and delete tenants, assign users to tenants, and view tenant-level stats and admin info. New screens live under glue-composer/src/app/admin/:

Tenants

List, create, and delete tenants (slug + display name).

Members

Assign users to a tenant and set their per-tenant roles.

Stats

Page counts, publish status, and activity per tenant.

4Editor UX & guardrails

Scoping is invisible if the UI makes the active tenant obvious and hard to confuse. The composer header carries a persistent tenant switcher tinted with the tenant's brand colour, so an editor always knows whose site they are editing.

Tenant switcher

Always in the header, tinted with the tenant brand colour; switching re-scopes the whole workspace.

Mid-edit guardrail

Switching tenants with unsaved changes prompts a confirm dialog before discarding.

Per-tenant page list

The page list only ever shows the active tenant's content — no cross-tenant bleed.

Confirmations that name the tenant

Publish and delete dialogs restate the tenant name so a destructive action is never ambiguous.

Every list and picker has a clear empty state: no pages yet, no tenants assigned, or no members — each with the right next action for the user's role.

5Also fixed here

Two currently unguarded routes get locked down. PUT /api/webpageContent/[id] and /api/webpage/list today have no permission check and no tenant scoping — both gain requirePermission() plus the tenantId filter, closing an IDOR gap. Separately, the full-database snapshot endpoint (db-snapshot/create) is restricted to the Super User only, since it can read across every tenant.

Architecture · Application deployment

Deployment

One repository builds everything. glue-main is deployed once per tenant as its own AWS Amplify app — each with a TENANT_SLUG and its own domain — while a single, shared glue-composer app serves every tenant. Each build carries only its tenant's static content, which is a strong isolation boundary in itself.

Proposed — awaiting approval. Nothing implemented yet.

1 · Topology

The monorepo (npm workspaces) is the single source of truth. From it, Amplify builds two kinds of app: exactly one shared glue-composer, and one glue-main per tenant. Every glue-main app is an independent Amplify project with its own domain, environment, and CDN — but all of them come from the same commit of the same repo.

2 · One glue-main build per tenant

Each tenant's Amplify app is the same glue-main code with a different environment. The app sets TENANT_SLUG (plus per-tenant secret references) as environment variables, and the existing glue-main/amplify.yml is reused unchanged — the build reads the slug from the environment, so there is nothing tenant-specific in the build spec.

Amplify app · environment variables (per tenant)bash
# Set on each tenant's Amplify app — the ONLY thing that differs between deploys
TENANT_SLUG=acme
TENANT_DOMAIN=acme.com

# Per-tenant secret references (resolved at runtime from SSM / Secrets Manager)
ZOHO_SECRET_ARN=arn:aws:secretsmanager:us-east-1:123:secret:acme/zoho
DRAFT_SECRET_ARN=arn:aws:ssm:us-east-1:123:parameter/acme/draft
glue-main/amplify.yml — unchangedyaml
version: 1
frontend:
  phases:
    preBuild:
      commands:
        - npm ci
    build:
      commands:
        # reads process.env.TENANT_SLUG from the app's environment
        - npm run build --workspace glue-main
  artifacts:
    baseDirectory: glue-main/.next
    files:
      - '**/*'
New tenant = one templated deploy. Provision each glue-main app through IaC (a CDK or Terraform Amplify module parameterised by tenantSlug, its domain(s), and secret ARNs). Onboarding a tenant becomes a single parameter set applied by the pipeline — no code change, no hand-editing of the console.

3 · Resolving the tenant

A single helper, glue-main/src/lib/tenant.ts, decides which tenant a request belongs to. When the app is pinned (the normal per-tenant deploy), it resolves straight from process.env.TENANT_SLUG — that's the primary path. A Host-header fallback is kept in reserve for a possible future consolidated deploy where one app fronts many domains. The whole thing is wrapped in React's cache() so it is computed once per request.

glue-main/src/lib/tenant.tsts
import { cache } from 'react';
import { getTenantBySlug, getTenantByDomain } from '@clouda/db';

// Deduped per request. Pinned slug is primary; Host is a future-proofing fallback.
export const resolveTenant = cache(async (host?: string) => {
  const pinned = process.env.TENANT_SLUG;
  if (pinned) return await getTenantBySlug(pinned);

  // Fallback: only used by a hypothetical consolidated multi-domain deploy.
  if (host) return await getTenantByDomain(host);

  throw new Error('Cannot resolve tenant: no TENANT_SLUG and no host');
});

4 · Using the slug for data & config

The resolved tenant threads into every data call (so queries are scoped) and drives the configuration that is hardcoded today. Values move to one of two places: non-secret settings onto the tenant record, and secrets into AWS SSM Parameter Store / Secrets Manager keyed by slug.

Onto the tenant record (non-secret)

baseUrl · GTM id (GTM-KHHMJL8 today) · SEO / metadataBase · Zoho public IDs.

Into SSM / Secrets Manager

Draft preview secret · Zoho API keys · any credential — referenced by the ARNs set in the app environment.

Fixes a live bug. Today layout.tsx uses metadataBase: "clouda.com" while the shared constants say clouda.io. Reading tenant.baseUrl makes this correct per tenant and removes the mismatch.
glue-main/src/app/layout.tsx (sketch)tsx
import { headers } from 'next/headers';
import { resolveTenant } from '@/lib/tenant';

export async function generateMetadata() {
  const tenant = await resolveTenant();
  return { metadataBase: new URL(tenant.baseUrl) };  // was hardcoded "clouda.com"
}

export default async function RootLayout({ children }: { children: React.ReactNode }) {
  const tenant = await resolveTenant();
  const gtmId = tenant.analytics.gtmId;         // was "GTM-KHHMJL8"
  return (
    <html>
      <head><GoogleTagManager id={gtmId} /></head>
      <body>{children}</body>
    </html>
  );
}

5 · Static generation & fresh content

generateStaticParams scopes to the pinned tenant, so each build — and each tenant's CDN — holds only that tenant's HTML. Nothing from another tenant is ever compiled into the bundle, which makes the build itself a hard isolation boundary.

Because static output is frozen at build time, add ISR plus an on-demand POST /api/revalidate guarded by a per-tenant secret. When an editor publishes in the composer, it calls this route so the live site refreshes the affected paths without a full rebuild.

glue-main/src/app/api/revalidate/route.ts (sketch)ts
import { revalidatePath } from 'next/cache';
import { resolveTenant } from '@/lib/tenant';

export async function POST(req: Request) {
  const { path, secret } = await req.json();
  const tenant = await resolveTenant();

  // Per-tenant secret resolved from SSM by slug (a secret — not stored on the tenant record).
  const expected = await getSecret(`/${tenant.slug}/revalidate`);
  if (secret !== expected) {
    return new Response('Unauthorized', { status: 401 });
  }
  revalidatePath(path);
  return Response.json({ revalidated: true });
}
Net effect. Publishing in the shared composer updates the correct tenant's live site in seconds, each site stays a self-contained static bundle, and onboarding a new tenant is one parameterised deploy.

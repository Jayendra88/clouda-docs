Roadmap & Risks

The phased, MVP-first plan to reach multi-tenancy, the risks that come with it and how each is mitigated, and the verification that proves every step is safe. The guiding constraint throughout: www.clouda.io keeps behaving exactly as it does today.

Proposed — awaiting approval. Nothing implemented yet.
MVP first. Phases 0–4 build the foundation and take one real second tenant fully live on its own domain and theme. Phases 5–8 are the heavier admin, registry and hardening tooling — valuable, but deferred until the platform is proven multi-tenant end-to-end.

Phased rollout

Each phase is additive and reversible, and none changes what visitors see on the existing site until we intentionally cut over the second tenant. The MVP milestone is Phase 4.

  1. Data foundation & shared packages MVP Stand up packages/db and packages/components as workspaces, add the tenant-scoping repository (getTenantRepo) that forces a tenantId on every read and write, add tenantId-leading indexes, then run an additive, idempotent backfill that stamps every existing document with the default tenant (clouda).
  2. glue-main tenant resolution & scoped reads MVP Resolve the tenant from TENANT_SLUG at build/request time, route all content queries (getWebpageContentBySlug, getGlobalSection, getHomePage) through the scoped repository, and drive per-tenant config from the resolved slug.
  3. Theming foundation MVP Introduce the CSS-variable token layer and per-tenant theme files loaded by TENANT_SLUG, with the existing brand values wired in as the clouda defaults so nothing shifts visually.
  4. Composer: tenant context & access control MVP Add a tenant switcher and tenant context, make roles (super-admin/admin/editor/viewer) per-tenant, scope every CRUD operation to the active tenant, add the Super User admin, and close the currently unguarded API routes.
  5. MVP milestone — onboard a real second tenant MVP Deploy a second glue-main app on its own domain with its own TENANT_SLUG and theme, author and publish its pages in the shared composer, and confirm ISR on-demand revalidation updates it without a rebuild. This is the point at which the platform is genuinely multi-tenant.
  6. Component configuration Later Replace the monolithic schema with per-component *.section.json files aggregated at build time, adding a tenant field so the editor palette can vary per tenant.
  7. Component visibility & editor tooling Later Add a component-visibility admin, a searchable palette, and an in-composer theme editor so tenant admins can self-serve without a developer.
  8. Component theming waves Later Migrate components off hardcoded colours and the isLightTheame flag onto theme tokens, in reviewable waves, publishing between each so no tenant regresses.
  9. Backup & restore hardening Later Make snapshot, backup and restore operations per-tenant and access-controlled, replacing the current unscoped dump.

Risks & mitigations

Every risk below has an owned mitigation baked into the plan — most are structural (the code makes the unsafe path impossible) rather than procedural.

RiskImpactMitigation
Cross-tenant data leakage One tenant reads or writes another tenant's pages. Mandatory getTenantRepo that forces tenantId on every query; a lint/CI gate blocking raw collection access; tenantId-leading indexes; and per-tenant deploy isolation so a misconfigured slug can only ever see its own data.
Partial backfill hides pages Existing live pages disappear mid-migration. Backfill is additive and idempotent, indexes are created first, the run verifies a residual count of 0 untagged documents, and it is fully reversible.
SSG blow-up / stale cache Every tenant's pages rebuilt together, or stale content served after publish. Static generation is scoped per tenant, ISR keeps pages fresh, and on-demand revalidation updates a page immediately without a full rebuild.
Duplicated data-access drift Copied query logic diverges and one copy forgets the tenant filter. Eliminated at the source: a single packages/db owns all data access, enforced by the same lint rule.
Pre-existing security holes Unauthenticated edits and cross-tenant reads today. All fixed as part of Phase 3: guard the unguarded PUT /api/webpageContent/[id] and /api/webpage/list, scope the snapshot dump, replace the hardcoded draft secret, and resolve the .com/.io metadata mismatch.
Theming regression A component looks wrong after the token migration. Tokens ship with the current values as fallbacks, migration runs in reviewable waves, and each wave is published before the next begins.
Stale tenant context A write lands under the wrong tenant after a switch or a revoked membership. Re-validate tenant membership on every sensitive write, not just at switch time.
Per-tenant deploy ops cost Managing N Amplify apps by hand becomes error-prone. Per-tenant deployments are templated as infrastructure-as-code so a new tenant is a parameterised, repeatable provision.

Verification

The plan is only done when these checks pass. They are the acceptance criteria for the MVP milestone.

  • Isolation tests. Reads and updates from tenant A against tenant B return nothing and 404 on a cross-tenant _id; every insert is proven to stamp tenantId.
  • Lint gate. CI fails on any raw collection access or startMongo call made outside packages/db, so the scoped repository is the only data path.
  • Backfill dry-run. The dry-run reports 0 untagged documents remaining and confirms the clouda document counts are unchanged before and after.
  • glue-main parity. With TENANT_SLUG=clouda, the site renders byte-identically to today — proof the refactor is behaviour-preserving.
  • Second-tenant E2E. Build and publish a page for the new tenant; it renders on the new domain with its own theme, tenant 1 never appears anywhere in it, and on-demand revalidation updates it without a rebuild.
  • Auth. requirePermission returns the resolved tenantId, no cross-tenant access is possible, and the Super User can manage all tenants.
Definition of done for MVP. A real second tenant is live on its own domain and theme, the existing clouda site is provably unchanged, and every isolation, lint, parity, E2E and auth check above is green.
